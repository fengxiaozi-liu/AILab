# /specify 工作流

当你（Gemini Agent）收到用户输入的 `/specify <需求描述>` 指令时，你必须按照以下步骤从需求描述生成特性规格文档。

## 工作流触发条件
- 当用户表达有新的功能需求，需要结构化记录时
- 当用户给出含糊其辞的需求，需要你识别并追踪待澄清项（CQ）时
- 当用户要求补充或重新梳理已有的需求说明时

## 执行步骤

1. **获取项目上下文** 
   - 使用文件读取工具（如 `view_file`）阅读项目根目录下的 `README.md`，了解项目的基础概况。
   - 如果存在 `docs/*/architecture.md`，阅读它以了解项目的架构约束和已有模块。
   - 扫描一下 `specs/` 目录，检查是否已有重复的特性说明文档，避免重复工作。

2. **生成分支短名**
   - 深入理解用户的自然语言描述，提炼关键**动作**和**对象**。
   - 按照 "动词-名词"（例如 `user-auth`）的风格生成 2-3 个单词的分支短名。
   - 保留专业系统缩写（如 oauth2, api），采用全小写，单词间用 `-` 隔开。绝对避免泛化名称（如 `add-feature`）。

3. **需求提炼与分析**
   - 读取模板文件 `.gemini/templates/spec-template.md`。
   - 参照 `speckit-specify` 技能规范（见 `.gemini/skills/speckit-specify/SKILL.md` 的 [§需求提炼] 部分），将用户的需求拆解为四要素：Actor(角色)、Action(动作)、Data(数据/实体)、Constraints(约束与边界)。

4. **生成架构与业务交叉的澄清问题 (CQ)**
   - 根据 `.gemini/skills/speckit-specify/SKILL.md` 中定义的 9 大评估维度（如：功能边界、异常/失败处理、外部依赖等），对第一版提炼的需求进行漏洞检查。
   - 列出所有信息缺失或含糊不清的点，并针对漏洞为你将生成的 Spec 结构添加待澄清问题（CQs）。
   - 切记：你绝对不可以凭空为用户做“合理假设”（比如默认重试 3 次）。如果不确定，必须抛出 CQ 追问用户！

5. **生成规格产物 (Spec)**
   - 根据以上的思考和分析结果，在 `specs/<分支短名>/spec.md` 路径下生成该需求文档。
   - 你只能创建符合 `.gemini/templates/spec-template.md` 结构的文档内容。

6. **自动推进与交互**
   - 将生成的 Spec 的核心逻辑向用户进行简要总结。
   - 汇报结论编排：
     - 若生成的 spec 中包含 `[待澄清]` 的 CQ（问题），请向用户**主动提问要求解答**（告诉用户这些 CQ 之后可以进入 `/clarify` 流程继续深度澄清）。
     - 若完全没有待澄清项，询问用户是否可以直接进入 `/plan` 流程进行技术方案设计。

## 核心写作原则（必须遵守）
- **聚焦业务价值**：只写"用户需要什么、为什么"，绝对不允许包含"用什么数据库什么包去实现"这种纯技术细节。
- **面向干系人**：它是需求沟通凭证，不写代码/框架/API 接口定义。
- **职责清晰**：Spec 是需求捕获工作台，不是最终的系统交付设计书。
