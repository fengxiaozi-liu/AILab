# /summary 工作流

当你在一个需求分支的所有的代码审查（code-review）结束后，或者用户需要一个最终存档总结文档。

## 适用场景
对目标特征功能分支的三要素（规格说明、实现计划、执行拆解列表），再加质量评估的最终代码审核结果进行提取的工具指令。 

## 执行步骤

1. **环境核验（严守纪律：不越级操作）**
   - 读取 `.gemini/skills/speckit-summary/SKILL.md` 的代码审查方法论。
   - 打开该功能文档集所在的文件夹。利用用户指定的特征目录名的 `$ARGUMENTS` 来定位（或者试图在你正在处理的项目 `specs/` 目录中找到它）。
   - 你需要定位下面全部四份文件：
     - `specs/<feature>/spec.md`
     - `specs/<feature>/plan.md`
     - `specs/<feature>/tasks.md`
     - `specs/<feature>/review.md`

2. **源数据收集压缩与只读检测**
   - 绝不允许修改前面提取到的这四份源码的任意内容。只能向其中提取或者阅读。
   - 解析出这四个里面缺失或残缺的情况（例如由于没做 `code-review`，没有找到 `review.md`，你不能强行停止生成，但是需要标注在最终摘要书中它的部分待执行）。如果全是缺失那么报告问题终止过程。

3. **格式化输出总览文件**
   - 使用 SKILL 填充规范对所有的信息做汇总整合：做了什么，怎么做的验证逻辑与阶段列表的收束结论，最后合并出 `specs/<feature>/summary.md` （如果有则覆盖重写）。
   - 只提供信息归档压缩（原事实汇总提取），不发表或者捏造针对源文档内容的主观评价。不判断业务合规与否，不要替原版去假设评价这功能的好坏，这些都不在总结范围内。

4. **汇报与继续流转**
   - 向用户发问，告诉他们文件的产出位置、主要结构和是否所有清单已成功结案即可。
