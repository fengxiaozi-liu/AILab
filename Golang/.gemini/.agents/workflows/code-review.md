# /code-review 工作流

当你听到用户调用 `/code-review` 时，充当验收环节的严格看门人。

## 适用场景
对已经在目标代码库内跑完的 implementation 产出代码，结合之前制定的规格与实现设计三件套执行**审计与报告**。

## 执行步骤

1. **环境核验与只读纪律（无权限改代码）**
   - 读取 `.gemini/skills/speckit-code-review/SKILL.md` 的代码审查方法论。
   - 打开包含当前进度的 `specs/<feature>/tasks.md`、`specs/<feature>/plan.md`。 如果这两个文档是不存在的，中止它，建议运行先前的预设步骤。并提醒用户 `/tasks` 全都得标记 `[x]` 后才能验收。如果不满，强制建议回去。

2. **四大评审维度验证（依赖 Kratos Pattern）**
   - 使用 `.gemini/skills/kratos-patterns` 框架与架构约束为尺度。
   - 对你代码库新产生/由于这一次特征修改所带来的增量文件做逐一比对：
     - **代码实现完成度（是否漏写？）**
     - **架构合规性（是否符合微服务分层规范要求？）**
     - **业务与接口性能缺陷？**
     - **安全准入？**
   - 只给报告与分析（写入 `specs/<feature>/review.md`）。
   - 你可以通过举例给出建议。

3. **汇总**
   - 绝不自动修改源码文件。
   - 产生的结果交给用户，如果有大量的负面评分，直接询问：我们是否用 `/implement` 去把安全或者合规问题 fix 并再继续。
