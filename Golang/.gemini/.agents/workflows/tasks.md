# /tasks 工作流

依据技术计划计划（plan.md），规划项目落地、生成编码实操（Kratos工作流顺序）阶段可执行任务列表。

## 适用场景
当你（Agent）接受完用户的业务核对，并且 `/plan` 生成的 `specs/<feature>/plan.md` 确定处于可用状态。

## 执行步骤

1. **基本约束检视与提取**
   - 读取该特性目录下的 `specs/<feature>/plan.md` 和 `specs/<feature>/spec.md`。如果计划文档不存在或者 spec 的 Status 取决于不为 Ready，提示用户需要先执行 `/plan` 或相应的需求准备步骤，并暂停运作。
   - 读取并提炼计划中所拆解阶段的任务要求特征，并匹配到对应的系统需求点（RQ，如果有的话）。

2. **按相级（Phase）分解编码实操步骤**
   - 依赖 `.gemini/skills/speckit-tasks/SKILL.md`，将该技术分解依据业务实操分解。
   - 这也是最核心的构建步骤：你必须严格遵循预先的 `kratos-patterns` （可在 `.gemini/skills/kratos-patterns` 查阅）以保持 **工作流顺序合理逻辑**（如要不要先建库、要不要先发协议层定义，再去到接口和组装层等步骤），设立拆分的 **阶段 Phase**。

3. **梳理任务列表**
   - 基于前序 Phase 分解，在每个阶段内，列出详细的**待实现任务节点**，例如：更新协议文件、运行 wire 生成依赖代码、写死单元测试用例。并且如果不能决定任务粒度，必须主动与用户进行询问。

4. **汇总校验输出**
   - 保存至该特性 `tasks.md` 到 `specs/<feature>/tasks.md` 下。
   - 任务清单文档需要遵循 `.gemini/templates/tasks-template.md` 模板的内容和要素格式。

5. **下一步编排建议**
   - 总结：该特性任务将执行几大主要模块、覆盖多少个原设定需求，统计摘要向用户明示。
   - 如果用户通过了计划任务，你可以推荐用户触发 **/analyze** 进行实施前的一致性查错扫描或开始做代码。
