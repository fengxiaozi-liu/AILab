# /implement 工作流

当你听到用户调用 `/implement` 命令时，启动实质性的代码编写过程。依据预定义的任务分解表，分阶段（Phase）在实际代码库中写代码！

## 适用场景
当你确定需求三件套已经准备好（并确保它们被加载），并确定用户给出了 `/implement` 指令时开始。

## 执行步骤

1. **环境准备与沟通确认**
   - 读取 `.gemini/skills/speckit-implement/SKILL.md` 的规范策略。
   - 读取该特性所有的相关文档三件套： `specs/<feature>/spec.md`，`specs/<feature>/plan.md` 以及 `specs/<feature>/tasks.md` 到你的分析环境中。任一不存在，立刻中止任务并在控制台建议用户运行相应的配置指令（例如 `/tasks` 等）。
   - **向用户询问他们的介入意愿**：“你是否想要在每个构建小阶段（Phase）结束时暂停并等你确认呢，还是一次全量跑完生成完毕后再核对？” （如果在执行本操作前用户已经做过批注，按用户批注的暂停规则走）。

2. **Phase 节点推进**
   - 依赖 `.gemini/skills/kratos-patterns` 等框架编码约束（包括模型、控制器等格式）。
   - 解析和分理 `tasks.md` 中的构建事项列表。判断是否已经有打过 `[x]` 标记的断点处——你可以**断点续做**（忽略过去打钩的）。
   - **按列表顺序在工程目标文件中实施这些代码！**

3. **阶段任务标记（检查表跟进）**
   - 在写完并保障每个拆分任务之后，立刻在本地实际的 `tasks.md` 内，在该节点行处用 `[x]` 在 checkbox 进行标记。（使用查找替换或者重新复写）。
   - 若遇到代码写到一半遇到无法逾越的依赖冲突或者缺少组件，立刻停下动作反馈报告，不再自动写。

4. **汇总校验输出**
   - 执行阶段如果在先前的暂停询问中，用户同意需要暂停确认，则写完一个 Phase 停下来，要求用户回复 "Continue" 再走向下一个 Phase。
   - 包含像生成器或者代码命令行依赖（如 wire 或 Ent 的 generate 操作），必须向系统提交并且自行确认没有报错，如果有错误停下重构或等待用户决策。

5. **全流程下一步建议**
   - 所有的代码落实、功能按计划都上线而且任务列表都打上了 `[x]`。
   - 提示：你可以推进 **/code-review** 完成质量大门的检查。
