# /clarify 工作流

这是 `/specify` （或 `/checklist` 给出的未通过预警）之后的深度澄清修复环节。

## 适用场景
当你发现 `specs/<feature>/spec.md` 中的 `待澄清问题（CQs）` 列表有待确定的问题，或者是用户调用 `/clarify` 指令并传达额外参数说明时启动。

## 执行步骤

1. **环境与问题扫描**
   - 读取 `.gemini/skills/speckit-clarify/SKILL.md` 了解问题覆盖度评估方法和提问原则。
   - 打开 `specs/<feature>/spec.md` 进行检查。如果不存在该文件，中止流程并建议运行 `/specify`。
   - 提取出待梳理的 CQ 问题队列，按照 9 大维度（功能边界、领域模型等）扫描该 Spec 当前的覆盖状态。

2. **单题提问互动 （非常重要：不要一次抛出全部问题）**
   - 虽然有多个问题，但为了保障效率，你必须**一题一题（一次一题）**向用户询问。
   - 每题呈现的内容包括：
     - 问题描述及其重要性。
     - **推荐答案与推荐理由（如果适用）**。
     - 包含选项给用户选（如 A, B, C 等）。
   - 在本回合暂停，等待用户的回答。用户回答（yes、A 或其它内容）后，进入下一步。
   - 如果用户在某一题中给出诸如 `stop`, `done`, `proceed` 等提前终止命令，立刻停止所有未提问的问题，尊重用户的结束意愿。

3. **实时增量回写与校验**
   - 用户每回答确定一题，你必须立刻将澄清的结果增量更新回写到原 `specs/<feature>/spec.md` 对应的该题区域。
   - 将这部分新增信息按规范要求校验。

4. **汇报与继续流转**
   - 在用户结束所有问题（或输入提前结案口令）以后：
     - 归纳一下“已澄清的”、“剩下的盲点”、“已忽略未确认的关键点”。
     - 如果全部分析完毕且无关键歧义（`Status` 置为 `Ready`），主动询问用户："无关键歧义，可继续。你想通过 **/checklist** 做一遍需求质量审查？还是由于已有信心，直接进入 **/plan** 进行技术方案设计？"
