# /checklist 工作流

当你听到用户调用 `/checklist` 命令时，你作为独立的质量审查工具执行以下流程。

## 适用场景
这是一个独立工具命令，可在规格定义（spec存在）后的任意阶段使用。检查的是 **需求质量**，而不是技术实现。

## 执行步骤

1. **环境与文件加载**
   - 检查项目上下文，确保目标特性的 `spec.md` 存在。如果不存在，立即终止运行并建议用户先执行 `/specify`。
   - 读取该特性目录下的 `spec.md`。

2. **识别检查范围（如有必要，提问澄清）**
   - 根据用户的 $ARGUMENTS（如无特别指定，则默认全量检查），识别出当前检查的领域。
   - 最多可以生成不超过 3 个动态前置澄清问题（参照 `.gemini/skills/speckit-checklist/SKILL.md` 的要求）。

3. **生成检查清单（Checklist）**
   - 使用 `.gemini/templates/checklist-template.md`（如有）作为基础参考。
   - 读取并运用 `.gemini/skills/speckit-checklist/SKILL.md` 中说明的规则，生成逐条检查的需求质量问句。
   - **绝对禁止包含实现验证语句**（如：“检查数据库是否建立了表X”），只能是需求质量质询（如：“是否规定了用户密码的最大重试次数？”）。
   - 至少 **50%** 的条目需要带上对原 spec 需求标签（如 RQ-001）的直接引用（追溯关联）。

4. **存储与汇报**
   - 将生成的清单存入 `specs/<feature>/checklists/` 目录下（同名追加或建立新文件）。
   - 向用户汇报生成的质量盲点与检查项。

5. **下一步流转**
   - 如果用户确认清单暴露出了大量未解决的严重需求缺陷，主动向用户提议：**转移到 `/clarify` 流程继续深度澄清需求**。
   - 如果检查清单顺利通过且无遗漏，主动询问用户是否准备好进入 `/plan` 流程。
