# /constitution 工作流

当你（Gemini Agent）收到用户输入的 `/constitution <原则更新或新建需求>` 指令时，你是本项目的宪法治理专员。

## 适用场景
为整个代码库确立系统或工程层面的最高级别约束，维护全局准则文档。

## 执行步骤

1. **寻址和预检查**
   - 检查 `.gemini/rules/project/` 下是否已经有过 `constitution.md` 或其他规范文件。
   - 打开根目录和核心规则文档等上下文，查看用户是否有明确指示或者系统已存在的规则说明需要做抽象层上升的。

2. **原则提炼（不落细则，只抓全局纲领）**
   - 读取 `.gemini/rules/project/*.md` 内容。如果你发现项目内确实缺少一部主文件名为 `constitution`，使用你的方法论，利用 `view_file` 或相关读取命令在系统已有信息的基础上（如用户要求：本项目一定且永远都要只用 Go 标准库等等），制定并创建项目专属的主宰宪法文件到规定库的相对目录： `.gemini/rules/project/constitution.md`。
   - 使用 `Principle-ID` 做每一条文准则的前端锚（比如： `[ENG_CONVENTION_01]` ）并指出它的细则依靠哪个 `Rule Source` 文件来进行具体的落地依据。

3. **规则下发和同步关联（确保不冲突）**
   - 在制定或修改原则后，确保不和下方的 `rules/*` 内容进行平行重复，或者如果有互相矛盾，自动将那些不符合规定的 `rules/*.md` 执行一致性更新。
   - 在 `constitution.md` 顶端建立一段自动记录修改影响的 HTML 注释报告（Sync Impact Report）。

4. **汇报与继续流转**
   - 生成本次立宪/修改文档对整个架构、环境代码（包括是否有任何违背了你的这些原则的东西，需要让接下来其他代码生成功能加以防范）的影响范围和原因。
   - 生成给用户的代办清单。
