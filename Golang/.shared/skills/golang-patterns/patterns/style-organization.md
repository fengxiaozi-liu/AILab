# Style Organization Pattern

## 这个主题解决什么问题

说明 Go 代码如何保持清晰的控制流、文件组织和可读性。

## 适用场景

- 日常函数实现
- 模块文件拆分
- 代码审查时统一风格

## 设计意图

风格与组织参考主要解释为什么稳定的文件和包结构会降低团队协作成本，而不只是罗列排版偏好。

- 组织方式稳定后，开发者更容易预判一个新类型、新函数该落在哪里。
- 文件布局和命名清楚时，检索、审查和重构成本都会下降。
- 风格约定如果能反映职责边界，就不再只是表面格式，而是结构化协作的一部分。

## 实施提示

- 先根据职责选择包和文件，再细化命名。
- 看到同类逻辑分散在多个文件里时，优先考虑按职责而不是按时间顺序整理。
- 如果一个文件同时混合模型、流程和基础设施细节，通常值得重新拆分。

## 推荐模式

- 早返回
- 短函数
- 命名稳定
- 同类型方法接收者风格一致

## 标准模板

```go
func Parse(s string) (int, error) {
    if s == "" {
        return 0, errors.New("empty")
    }
    return strconv.Atoi(s)
}
```

## 常见坑

- 多层嵌套
- 文件职责过大
- import alias 过多
