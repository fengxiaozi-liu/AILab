# Errors Pattern

## 这个主题解决什么问题

说明 Go 代码中错误值如何传递、包装、判断和补上下文。

## 适用场景

- 编写普通 Go 错误处理逻辑
- 需要补上下文
- 需要 `errors.Is/As` 判断

## 设计意图

错误模式的重点是让 Go 代码里的错误既能保留上下文，又能保持调用链上的语义清晰。

- 理解错误是携带语义的值后，更容易在包装、透传和判断之间做稳定选择。
- 清晰错误模式能减少日志里难以定位的匿名字符串错误。
- 错误写法稳定后，调用方也更容易判断哪些错误适合继续向上处理。

## 实施提示

- 先判断错误是要补上下文、做语义转换，还是原样透传。
- 再选择 `fmt.Errorf`、自定义类型或项目已有包装方式。
- 如果错误最终会暴露到协议层，顺手考虑它与项目 error 体系的衔接。

## 推荐模式

- 早返回
- 需要额外上下文时使用 `%w`
- 在调用方用 `errors.Is/As` 判断

## 标准模板

```go
v, err := load(id)
if err != nil {
    return "", fmt.Errorf("load user %d: %w", id, err)
}
```

## Good Example

```go
if err := do(ctx); err != nil {
    return fmt.Errorf("do task: %w", err)
}
```

## 常见坑

- 丢失原始错误链
- 错误上下文过少，难以排查
- 调用方无法区分错误类型
