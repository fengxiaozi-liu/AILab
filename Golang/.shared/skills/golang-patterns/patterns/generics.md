# Generics Pattern

## 这个主题解决什么问题

说明 Go 泛型适合用在什么场景，以及如何写出可读的通用结构或函数。

## 适用场景

- 容器工具
- 通用转换逻辑
- 重复出现的类型安全模式

## 设计意图

泛型模式的重点是解释什么时候值得抽出类型参数，而不是把所有重复代码都推进泛型层。

- 适度泛型可以减少重复和类型断言，过度泛型会让调用和调试成本上升。
- 理解这一点后，更容易区分通用容器能力和具体业务逻辑。
- 类型参数设计清楚时，函数签名本身就能传达约束和复用范围。

## 实施提示

- 先确认重复点是算法结构重复，还是只是两个业务函数长得像。
- 再决定类型参数、约束和返回值形态。
- 如果调用方为了使用泛型写了更多样板代码，通常值得回头评估抽象收益。

## 推荐模式

- 在工具层使用泛型
- 用尽量小的约束表达能力

## 标准模板

```go
func MapSlice[T any, R any](in []T, fn func(T) R) []R {
    out := make([]R, 0, len(in))
    for _, item := range in {
        out = append(out, fn(item))
    }
    return out
}
```

## 常见坑

- 为单点业务引入复杂类型参数
- 约束过宽，失去表达力
