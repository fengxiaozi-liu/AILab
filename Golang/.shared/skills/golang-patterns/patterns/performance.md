# Performance Pattern

## 这个主题解决什么问题

说明 Go 代码中常见的低风险性能优化点，以及何时值得关注分配和热路径。

## 适用场景

- 高并发路径
- 大量切片/map 构造
- 重复对象创建

## 设计意图

性能模式参考主要帮助理解优化是围绕热点和证据展开的，而不是把所有代码都写成微优化风格。

- 先找到真正热点，才能决定是否需要调整分配、缓存、批量化或并发策略。
- 可读性和性能通常需要平衡，解释层能帮助避免无证据的过早优化。
- 当性能手法与业务结构绑定时，后续维护者也更容易理解优化为什么存在。

## 实施提示

- 先判断瓶颈来自 CPU、内存、锁竞争还是 I/O。
- 再选择最贴近问题的优化手段，而不是同时套多种技巧。
- 如果优化让代码明显更难读，优先留下足够说明它在解决什么热点问题。

## 推荐模式

- 预分配 slice/map
- 避免热路径重复分配
- 先测量，再优化

## 标准模板

```go
out := make([]int, 0, len(in))
for _, v := range in {
    out = append(out, v)
}
```

## 常见坑

- 为了微优化牺牲可读性
- 没有热点证据就过度优化
