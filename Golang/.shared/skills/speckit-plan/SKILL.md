# Spec Kit Plan Skill

## 何时使用

- 需要生成技术实施计划（plan.md）时使用
- 若已有 spec，优先基于 spec 生成；若 spec 信息不足，则生成可执行草案并显式列出待澄清项

## 输入

- 必需：用户输入（目标、范围、约束、预期交付物）
- 可选：`specs/<feature>/spec.md`
- 模板：`./templates/plan-template.md`

## 依赖 Skill

- 项目适配器 Skill：按当前项目类型加载对应适配器
- 例如 Kratos 项目加载 `kratos-patterns`
- `speckit-plan` 只负责标准化规划流程，不负责项目判定

## 工作流

### 1. 前置检查

1. 读取 `./templates/plan-template.md`
2. 若存在 `specs/<feature>/spec.md`，提取 RQ、关键实体、约束与边界
3. 若 spec 不存在或信息不足，继续生成草案，但把缺失点列为待澄清项

### 2. 项目适配输入

明确项目结构并加载项目skill

### 3. Phase 0 领域调研

1. 分析关键实体及关系
2. 分析状态流转与异常路径
3. 识别外部依赖与上下游服务
4. 每个调研项输出：
   - 决策
   - 理由
   - 排除方案

### 4. Phase 1 技术设计

根据项目类型输出设计要点：

- 抽象定义/基础协议项目：
  - 契约、模型、共享能力、对外接口
- 业务实现项目：
  - 核心实体、应用层、数据层、对外接口、测试与验证
- 网关/接入项目：
  - 路由、代理、协议适配、依赖边界

### 5. 项目适配补充

若已加载项目适配器 skill，则用适配器知识补充：

1. 框架或平台层面的特有约束
2. 关键设计决策的规范来源
3. 与项目规范可能冲突的风险点

### 6. 写入规则

- 写入路径：`specs/<feature>/plan.md`
- 保持模板章节结构
- 若 spec 存在，Phase 0/1 内容应可追溯到 RQ
- 若 spec 不存在，则显式标注临时需求点与待澄清项

### 7. 质量校验

| # | 校验项 | 标准 |
|---|--------|------|
| 1 | 需求覆盖 | 每条 RQ 或需求点至少映射到一个设计项 |
| 2 | 实体完整 | 关键实体与关系已分析 |
| 3 | 交付物明确 | 每阶段有明确产出物 |
| 4 | 依赖已识别 | 关键依赖与约束已显式说明 |
| 5 | 风险可验证 | 关键风险有验证或缓解策略 |

## 输出

- `specs/<feature>/plan.md`
