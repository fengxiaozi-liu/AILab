# Comment Rule

## Principles

- 注释默认短小，一般用一行说明功能、规则或关键语义。
- 注释只放在关键位置，帮助理解边界、阶段和不直观的业务含义。

## Specification

- 对外协议、领域对象、UseCase、Repo、Service、ProviderSet、Ent schema 都应保留稳定的主题注释。
- 导出类型、导出函数、导出方法优先使用“对象/能力 + 作用”式注释，而不是重复函数名。
- 注释长度优先控制在一行内，只有确有必要时才扩展为多行说明。
- 关键位置的阶段切换、复杂分支、流程收口点允许补充短注释。
- 注释默认优先使用中文，只有外部接口名、标准术语或英文语义更稳定时才保留英文。
- Ent schema、field、领域对象、UseCase、Repo、Service 应补充能表达业务语义的注释。
- 表注释、字段注释、枚举描述应直接表达业务含义、状态含义或展示语义。
- 大块逻辑分段允许使用章节注释，但只用于明显阶段边界。

## Prohibit

- 禁止只写无业务含义的空泛注释，例如“处理数据”“执行逻辑”。
- 禁止把每一小段代码都加注释，导致正文被注释淹没。
- 禁止把过期实现细节、临时排障说明长期保留在正式注释中。
- 禁止在生成物上手改注释。
- 禁止注释与当前命名、状态语义、字段用途不一致。
- 禁止在 proto 文件中添加任何说明性注释、分隔线、章节线或装饰性注释。

## Checklist

- 注释是否解释了业务语义，而不是重复代码字面意思？
- Proto / Ent schema 是否补齐了表意注释？
- 是否避免在生成物中手改注释？
