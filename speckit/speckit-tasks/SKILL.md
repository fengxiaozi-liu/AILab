# Spec Kit Tasks Skill

## 何时使用

- 需要把计划拆成可执行任务时使用
- plan.md 存在时优先基于 plan 生成；否则基于用户输入和可选 spec 生成最小任务清单

## 输入

- 必需：用户输入（范围、约束、交付物）
- 可选：`specs/<feature>/plan.md`
- 可选：`specs/<feature>/spec.md`
- 模板：`./templates/tasks-template.md`

## 依赖 Skill

- agent 先识别当前项目类型，再决定是否加载项目适配 skill 或语言 skill
- 若 agent 已识别到项目约束，则将对应规则作为任务拆解补充输入
- `speckit-tasks` 只负责标准化任务拆解，不负责静态绑定具体项目 skill

## 工作流

### 1. 前置检查

1. 读取 `./templates/tasks-template.md`
2. 若存在 spec，提取 RQ 编号列表
3. 若存在 plan，提取项目类型、设计要点、风险项
4. 识别当前项目类型与结构；若已识别到项目约束，则一并纳入输入
5. 若 plan 不存在，则生成最小任务清单并显式标注缺失设计信息

### 2. 上下文提取

从 plan 中提取：

- 项目类型
- 阶段性设计结果
- 关键交付物
- 风险与依赖

从 spec 中提取：

- RQ 列表
- 关键实体
- 约束与边界

### 3. Phase 划分

- 若 agent 已识别到项目约束，则可按项目规则补充调整 Phase 顺序
- 默认按以下最小通用顺序划分：
  - 定义契约/模型
  - 核心实现
  - 接入层
  - 生成/装配
  - 测试/验证
  - 收尾

### 4. 任务生成

1. 每个任务应可独立执行、可验证
2. 每个任务应尽量标注对应 RQ
3. 同一 Phase 内无依赖冲突的任务可标记 `[P]`
4. 每个任务应包含精确文件路径

### 5. 路径约定

- 优先使用当前项目真实路径
- 若未识别到项目约束，则根据仓库事实推导路径

### 6. 任务格式

统一格式：

```md
- [ ] T001 [P] [RQ-001] 描述 + 文件路径
```

规则：

- `TaskID` 全文唯一
- `[P]` 仅用于可并行任务
- `[RQ-xxx]` 用于需求追踪；无 spec 时可用临时编号

### 7. 写入规则

- 写入路径：`specs/<feature>/tasks.md`
- 使用模板章节结构
- 若 agent 已识别到项目约束，Phase 顺序应与项目规则一致
- 否则使用最小通用顺序

### 8. 质量校验

| # | 校验项 | 标准 |
|---|--------|------|
| 1 | 需求覆盖 | 每条 RQ 或需求点至少被一个任务引用 |
| 2 | Phase 顺序正确 | 若存在项目约束则与项目规则一致，否则通用顺序自洽 |
| 3 | 依赖无环 | 后置任务不被前置 Phase 反向依赖 |
| 4 | 路径准确 | 文件路径符合当前项目结构 |
| 5 | 粒度合理 | 每个任务可在一次实现中完成 |

## 输出

- `specs/<feature>/tasks.md`
