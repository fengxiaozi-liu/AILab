# Spec Kit Plan Skill

## 何时使用

- 需要生成技术实施计划 `plan.md`
- 已有 `spec.md`，需要把需求转成阶段化计划
- 没有完整 spec，但需要先形成一份可执行的规划草案

## 输入

- 必需：用户输入，包括目标、范围、约束、预期交付物
- 可选：`specs/<feature>/spec.md`
- 模板：`./templates/plan-template.md`

## 职责边界

- `speckit-plan` 负责通用规划流程和输出结构
- 当前项目是什么类型，由 agent 在运行时识别
- 是否需要加载项目适配 skill，由 agent 在运行时决定
- 本 skill 不点名任何具体框架或项目 skill

## 工作流

### 1. 前置检查

1. 读取 `./templates/plan-template.md`
2. 如果存在 `specs/<feature>/spec.md`，提取需求、关键实体、约束和边界
3. 如果 spec 不存在或信息不足，继续生成草案，但显式列出待澄清项

### 2. 项目识别输入

agent 需要先结合仓库事实识别当前项目类型和结构。

可识别的通用类型示例：

- 服务端项目
- 库 / SDK 项目
- CLI 工具
- 前端应用
- 网关 / 接入层项目
- 数据处理 / 批处理项目

如果 agent 已识别到项目适配 skill，则将该适配 skill 的约束作为规划输入。  
如果暂时无法识别项目类型，则退化为通用规划流程，并显式记录不确定项与假设。

### 3. Phase 0 领域调研

1. 分析关键实体及关系
2. 分析状态流转与异常路径
3. 识别外部依赖与上下游服务
4. 每个调研项输出：
   - 决策
   - 理由
   - 排除方案

### 4. Phase 1 技术设计

根据当前项目类型输出设计要点。

通用设计视角示例：

- 服务端项目：核心实体、接口边界、数据访问、测试与验证
- 库 / SDK 项目：公共接口、兼容性、依赖边界、示例与测试
- CLI 工具：命令结构、参数输入、输出格式、错误处理、执行流程
- 前端应用：页面结构、状态管理、接口交互、组件边界、测试策略
- 网关 / 接入层项目：路由、代理、协议适配、依赖边界
- 数据处理项目：输入输出格式、批处理流程、容错策略、监控与验证

### 5. 项目适配补充

如果 agent 在运行时已加载项目适配 skill，可补充：

1. 框架或平台特有约束
2. 关键设计决策的规范来源
3. 与项目现有规范可能冲突的风险点

如果没有适配 skill，则继续按通用规划流程输出，不阻塞 plan 生成。

### 6. 写入规则

- 输出路径：`specs/<feature>/plan.md`
- 保持模板章节结构
- 如果 spec 存在，Phase 0 / Phase 1 内容应可追溯到需求点
- 如果 spec 不存在，则显式标注临时需求点和待澄清项

### 7. 质量校验

| # | 校验项 | 标准 |
|---|--------|------|
| 1 | 需求覆盖 | 每条需求或需求点至少映射到一个设计项 |
| 2 | 实体完整 | 关键实体与关系已分析 |
| 3 | 交付明确 | 每个阶段有明确产出物 |
| 4 | 依赖识别 | 关键依赖与约束已显式说明 |
| 5 | 风险可验证 | 关键风险有验证或缓解策略 |
| 6 | 假设可见 | 若项目类型识别不确定，假设与待澄清项已显式记录 |

## 输出

- `specs/<feature>/plan.md`
