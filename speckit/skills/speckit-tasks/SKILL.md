# Spec Kit Tasks Skill

## 何时使用

- 需要把计划拆成可执行任务时使用
- `plan.md` 存在时优先基于 plan 生成；否则基于用户相关输入和可选 spec 生成最小任务清单

## 输入

- 必需：用户输入（范围、约束、交付物）
- 可选：`specs/<feature>/plan.md`
- 可选：`specs/<feature>/spec.md`
- 模板：`./templates/tasks-template.md`（工作流开始前优先读取，输出尽量沿用其章节结构与列表格式）

## 职责边界与依赖

- agent 先识别当前项目类型，决定在拆解时是否需要遵守额外的架构或语言规范。
- `speckit-tasks` 负责输出具备清晰组织结构与技术标签的 `tasks.md`，它直接服务于后续的 `speckit-implement` 阶段。
- 应尽量让任务清单既能在物理上低冲突执行，又能在同一 Phase 内保持技术栈和认知连贯性。

## 工作流

### 1. 前置检查

1. **读取 `./templates/tasks-template.md`**，将其作为后续输出的主要格式基准。
2. 在输出前明确说明本次 `tasks.md` 将基于 `./templates/tasks-template.md` 生成。
3. 按模板既有章节和任务格式组织内容，再填入任务拆解结果。
4. 获取当前需求的基础上下文。
5. 若存在 spec，提取 RQ（需求追踪）编号列表。
6. 若存在 plan，提取总体阶段划分、架构设计要点与潜在风险项。
7. 识别当前项目类型；若有特定项目约束，将其纳为任务排布的基础准则。
8. 若没有任何文档依据，则基于最原始输入推导最小任务清单，并附上因信息缺失可能导致的风险标注。

### 2. 上下文与依赖提取

从已有材料中提纯：

- **核心数据流与调用链路**：哪些模块必须优先完成以充当底层基座？
- **物理阻塞节点**（如：表结构 -> ORM 实体 -> 核心业务服务 -> API 路由 -> 前端对接）
- **技术栈交界处**：明确区分哪些任务属于 DB 层，哪些是纯逻辑层，哪些是视图/UI 改动。

### 3. Phase 的科学划分原则

Phase 的存在不仅仅是为了串行防冲突，也是在为后续 `speckit-implement` 提供更低认知负荷、更高内聚的实施上下文。

通常不要把所有无依赖任务都塞进一个超大的 Phase。划分 Phase 时优先兼顾以下几点：

1. **优先隔离强依赖**：上游基座缺失时，不应过早开展中游工作；存在明确时序依赖时，优先拆分为前后 Phase。
2. **优先保持技术栈高内聚**：即使互不阻碍，也可考虑把后端建模、接口、前端视图拆成不同 Phase，减少下游执行时的上下文撕裂。
3. **控制单个 Phase 容量**：当一个 Phase 中的平级任务过多时，可按业务模块或技术切面继续细分。

### 4. 任务生成粒度与排布

1. **粒度控制**：每个任务应当做到“微架构原子化”。不应过于宏大（如“实现某业务模块全流程”），也不应细碎如行级代码（如“修改某业务逻辑文件第 5 行”），应定位到功能函数、数据模型、独立组件或特定配置的闭环级别（如“在核心逻辑文件中新增创建订单的方法实体”）。
2. **扁平排布**：在具体的某一个 Phase 内，只需通过列表把该阶段任务列出即可。
3. **精准路径与意图标注**：给每个任务显式加上技术切面（技术雷达）标识，便于下游执行器决定加载何种特定领域的知识库或动作。

### 5. 任务记录格式

所有任务行的格式以 `./templates/tasks-template.md` 中的示例为唯一基准，输出时严格照搬，无需在此重复定义。

### 6. 写入规则

- 写入目标文件：`specs/<feature>/tasks.md`
- 以 `./templates/tasks-template.md` 作为结构骨架，将占位符替换为本次推导的真实内容；如有需要可在章节内部追加，但尽量不破坏模板主体结构。
- 结构采用自顶向下的 Phase 瀑布流，各阶段天然自带前置顺位依赖。

### 7. 质量校验维度

| # | 校验项 | 标准 |
|---|--------|------|
| 1 | 需求覆盖无死角 | 每条需求点（RQ）至少能映射到一个实体的 TaskID |
| 2 | 技术域解耦 | 跨度极大的不同技术栈任务是否被有意识地分配到了独立的 Phase 中？ |
| 3 | 防爆栈控制 | 单一 Phase 内的任务数量是否过多？是否做到了适时地再切分？ |
| 4 | 上下文自洽 | 任务间的逻辑先后顺位是否合理（如确保数据先行、后跟接口、最后视图）？ |
| 5 | 定位精确 | 是否提供了充分的相对路径指引或组件目标标识？ |

## 输出

- `specs/<feature>/tasks.md`
- 完成后提示用户主动复查 `tasks.md`，确认无误后衔接执行 `/implement`。
