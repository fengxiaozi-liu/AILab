# Spec Kit Tasks Skill

## 何时使用

- 需要把计划拆成可执行任务时使用
- `plan.md` 存在时优先基于 plan 生成；否则基于用户相关输入和可选 spec 生成最小任务清单

## 输入

- 必需：用户输入（范围、约束、交付物）
- 可选：`specs/<feature>/plan.md`
- 可选：`specs/<feature>/spec.md`
- 模板：`./templates/tasks-template.md`

## 职责边界与依赖

- agent 先识别当前项目类型，决定在拆解时是否需遵守指定的架构或语言规范。
- `speckit-tasks` 负责输出具备清晰组织结构与技术标签的 `tasks.md`，它直接服务于后续的 `speckit-implement` 阶段。
- 必须确保所生成的任务清单既能在物理上无冲突串行执行，又能在同一 Phase 内保持技术栈和认知的连贯性。

## 工作流

### 1. 前置检查

1. 获取当前需求的基础上下文。
2. 若存在 spec，提取 RQ（需求追踪）编号列表。
3. 若存在 plan，提取总体阶段划分、架构设计要点与潜在风险项。
4. 识别当前项目类型；若有特定项目约束，将其纳为任务排布的基础准则。
5. 若没有任何文档依据，则基于最原始输入推导最小任务清单，并附上因信息缺失可能导致的风险标注。

### 2. 上下文与依赖提取

从已有材料中提纯：

- **核心数据流与调用链路**：哪些模块必须优先完成以充当底层基座？
- **物理阻塞节点**（如：表结构 -> ORM 实体 -> 核心业务服务 -> API 路由 -> 前端对接）
- **技术栈交界处**：明确区分哪些任务属于 DB 层，哪些是纯逻辑层，哪些是视图/UI 改动。

### 3. Phase 的科学划分原则

Phase 的存在不仅仅是为了串行防冲突，更是为了给后续 `speckit-implement` 框定“低认知负荷”、“高内聚执行”的实施上下文。

绝不能盲目把所有无依赖的任务全部塞进一个超大的 Phase 里。划分 Phase 时必须同时兼顾以下三点：

1. **强依赖必须隔离**：上游基座缺失时不可开展中游工作，存在时序依赖的必须拆分为首尾 Phase。
2. **技术栈高内聚与边界识别**：即使互不阻碍，也强烈建议把**后端建表/模型**与**前端组件**分离为不同 Phase（例如 `Phase 1: 数据模型与服务层`、`Phase 2: 接口与路由`、`Phase 3: 视图层`）。这能让 implement 运行时在同一个 Phase 内只用加载并聚焦极少量的 Skill，大幅降低 AI 幻觉和精力撕裂。
3. **适度的任务容量控制**：当一个 Phase 中容纳的平级子项过多（比如超过 10 个独立修改的组件），即便它们性质相同也应按照业务模块切分成 `Phase 1a`, `Phase 1b` 等独立阶段，防止单次执行上下文过热崩盘。

### 4. 任务生成粒度与排布

1. **粒度控制**：每个任务应当做到“微架构原子化”。不应过于宏大（如“实现某业务模块全流程”），也不应细碎如行级代码（如“修改某业务逻辑文件第 5 行”），应定位到功能函数、数据模型、独立组件或特定配置的闭环级别（如“在核心逻辑文件中新增创建订单的方法实体”）。
2. **扁平排布**：在具体的某一个 Phase 内，只需通过列表把该阶段任务列出即可。
3. **精准路径与意图标注**：给每个任务显式加上技术切面（技术雷达）标识，便于下游执行器决定加载何种特定领域的知识库或动作。

### 5. 任务记录格式

必须采用统一的格式标签体系，在任务首部加上 **`[技术域]`** 等意图标识。

```md
- [ ] T001 [Backend/API] [RQ-001] 新增路由控制器 `src/controllers/user.ts` 以响应注册请求
- [ ] T002 [DB/Schema] [RQ-002] ...
- [ ] T003 [Frontend/UI] [RQ-003] ...
```

格式约定：

- `TaskID`：全局递增唯一（T001, T002...）
- `[技术域/标签]`：概括该任务属于哪一层架构或影响面（如 `[Database]`, `[Core Service]`, `[UI Component]`, `[Config]`），帮助后续 Agent 识别预加载策略。
- `[RQ-xxx]`：用于需求双向追踪，无法对应时使用临时功能点编号。
- 任务描述最后应附上能够明确指向的具体文件路径或基准包名。

### 6. 路径推导

- 优先使用当前项目内已存在的真实目录和常见范式。
- 遇到新建模块，必须参考当前仓库风格，使用具备常识性指导的可执行路径

### 7. 写入规则

- 写入目标文件：`specs/<feature>/tasks.md`
- 结构需采用自顶向下的 Phase 瀑布流，各阶段天然自带前置顺位依赖。
- 首部不再输出复杂的 Mermaid 图，用几句话简单说明核心依赖和推进顺序即可。

### 8. 质量校验维度

| # | 校验项 | 标准 |
|---|--------|------|
| 1 | 需求覆盖无死角 | 每条需求点（RQ）至少能映射到一个实体的 TaskID |
| 2 | 技术域解耦 | 跨度极大的不同技术栈任务是否被有意识地分配到了独立的 Phase 中？ |
| 3 | 防爆栈控制 | 单一 Phase 内的任务数量是否过多？是否做到了适时地再切分？ |
| 4 | 上下文自洽 | 任务间的逻辑先后顺位是否合理（如确保数据先行、后跟接口、最后视图）？ |
| 5 | 定位精确 | 是否提供了充分的相对路径指引或组件目标标识？ |

## 输出动作

- 生成并把内容覆写更新到 `specs/<feature>/tasks.md`
- 写入成功后，提示用户主动复查一遍 `tasks.md`，等待确认无误之后，方可提示其衔接继续执行 `/implement` 。
