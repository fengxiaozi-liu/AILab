# Spec Kit Clarify Skill

## 何时使用

- `spec.md` 已存在，且其中包含待回答的 CQ
- 用户刚回答了一批 CQ，需要把答案回写进 spec
- 没有 spec，但需要先做一轮澄清再交给 `/specify`

## 职责边界

- `speckit-clarify` 负责**回答并消化 CQ**
- 可以补充少量新的 CQ，但仅限回答过程中暴露出的必要问题
- 不负责做系统化需求审计；那是 `/checklist` 的职责
- 不依赖 `checklists/*.md` 作为输入来源，唯一权威澄清载体是 `spec.md` 中的 CQ

## 输入

- 可选：`specs/<feature>/spec.md`
- 必需：用户给出的回答、约束或补充说明

## 工作流

### 1. 模式判定

本技能支持两种模式：

1. **Spec 澄清模式**
   - 条件：存在 `specs/<feature>/spec.md`
   - 目标：解决文档中已有的 `[待澄清]` CQ，并把结论同步回需求描述区
2. **预澄清模式**
   - 条件：spec 尚不存在
   - 目标：把用户补充信息收敛成“澄清结论摘要”，交给 `/specify`

### 2. 读取与筛选

在 Spec 澄清模式下：

- 读取 `spec.md`
- 找出所有 `A (澄清结论)` 仍为 `[待澄清]` 的 CQ
- 优先处理对架构、建模、边界、验收最关键的问题
- 已有明确答案的不重复追问

### 3. 澄清与回写

每获得一个有效答案，就执行：

1. 将答案写入对应 CQ 的 `A (澄清结论)`
2. 同步更新需求描述区的相关章节
3. 若与旧表述冲突，直接替换旧表述，不做并存堆叠
4. 仅当答案暴露出新的关键缺口时，追加新的 CQ

在预澄清模式下：

- 不创建 spec
- 输出结构化“澄清结论摘要”
- 明确哪些问题已收敛，哪些仍待决定

### 4. 状态规则

- 只要仍存在任意 `[待澄清]` CQ，`spec.md` 状态必须为 `Clarifying`
- 当所有 CQ 都已有结论且正文已同步完成时，状态改为 `Ready`

### 5. 写后校验

每次回写后检查：

- CQ 与答案一一对应，无重复
- 需求描述区与澄清结论一致
- 文档中不存在已回答但仍标记 `[待澄清]` 的项目
- Markdown 结构保持有效

## 输出

- Spec 澄清模式：更新后的 `specs/<feature>/spec.md`
- 预澄清模式：对话内的“澄清结论摘要”
- 若仍有未解 CQ，提示继续 `/clarify`
- 若已全部澄清完成并状态为 `Ready`，提示先运行 `/checklist`，通过后再进入 `/plan`
