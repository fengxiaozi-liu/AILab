# Spec Kit Specify Skill

## 何时使用

- 用户给出自然语言需求，希望生成/更新 spec。

## 输入

- 用户需求描述

## 工作流

### §上下文获取

1. 读取当前项目下对应的 `README.md` 文档了解项目概况。
2. 读取 `docs/*/architecture.md` 文档了解架构约束与已有模块。
3. 若存在相关 spec 历史（`specs/` 目录），扫描已有特性避免重复。

### §短名生成

从用户描述生成 2-3 词分支短名：

- 从描述提炼关键**动作**和**对象**
- 优先"动词-名词"风格（如 `user-auth`、`order-export`）
- 保留领域术语/缩写（OAuth2、API、JWT、RBAC）
- 全小写，单词间用 `-` 连接
- 避免泛化词（如 `add-feature`、`update-thing`）

### §需求提炼

1. 读取 `./templates/spec-template.md` 获取模板结构。
2. 从用户描述中抽取四要素，映射到模板章节：

| 要素 | 说明 | 对应模板章节 |
|------|------|-------------|
| **Actor** | 谁在使用（用户角色） | 目标用户 |
| **Action** | 要做什么（核心行为） | 核心需求 |
| **Data** | 涉及哪些数据/实体 | 关键实体 |
| **Constraints** | 限制条件（权限、频率、容量等） | 约束与边界 |

3. 填充规则：
   - **背景与动机**：从用户描述中提炼"为什么"
   - **目标用户**：明确列出涉及的角色
   - **核心需求**（RQ 编号）：每条一句话，可测试、无歧义
   - **关键实体**：只描述"是什么"，不描述"怎么存"；无则标"不适用"
   - **约束与边界**：权限、容量、异常场景等

### §澄清问题生成

1. 按 9 大评估维度扫描提炼结果，识别**信息缺失或模糊的点**：

| # | 评估维度 | 扫描要点 |
|---|---------|----------|
| 1 | 功能范围与行为 | 功能边界是否明确、行为是否完整 |
| 2 | 领域与数据模型 | 实体/关系是否清晰、字段是否确定 |
| 3 | 交互与 UX 流程 | 用户操作路径是否完整（后端项目通常不涉及） |
| 4 | 非功能属性 | 性能/可靠性/可观测/安全/合规要求是否明确 |
| 5 | 外部集成依赖 | 上下游系统/第三方服务是否已识别 |
| 6 | 边界与失败处理 | 异常/极端/并发场景是否覆盖 |
| 7 | 约束与权衡 | 技术/业务限制是否已声明 |
| 8 | 术语一致性 | 关键概念是否有歧义或多义 |
| 9 | 完成信号 | 可测验收条件是否可推导 |

2. 仅对当前特性**实际涉及的维度**生成 CQ，不涉及的维度跳过。
3. 每个 CQ 条目：
   - 编号：CQ-001, CQ-002, ...
   - 类别：对应的评估维度名称
   - 详细描述：为什么需要澄清
   - 可选项：列出 2-4 个合理选项供用户选择（如适用）
   - 澄清结果：初始为 `[待澄清]`
4. 约束：
   - **不做合理默认**，必须追问
   - 不限制 CQ 数量，根据实际状况列出不确定/模糊的CQ
   - 优先生成影响架构、建模、测试设计的维度问题

### §写入规则

- 写入路径：`specs/<feature>/spec.md`（`<feature>` = 短名）
- **保持模板两大区块结构**：需求描述 + 待澄清问题
- 子章节可按实际内容增减，但两大区块不可删除或调换

### §质量校验

1. 生成校验清单：`specs/<feature>/checklists/requirements.md`
2. 校验项：

| # | 校验项 | 标准 |
|---|--------|------|
| 1 | 无实现细节 | 需求描述不含代码/框架/API/数据库等技术方案 |
| 2 | 需求可测试 | 每条 RQ 都能推导出验收条件 |
| 3 | 无歧义 | 无"显然"、"合适的"、"必要时"等模糊措辞 |
| 4 | 四要素完整 | Actor/Action/Data/Constraints 均已识别或标注不适用 |
| 5 | 边界场景覆盖 | 空值、极端量、并发、异常路径有提及 |
| 6 | 澄清项闭环 | 所有 CQ 均有澄清结果（或文档 Status 仍为 Clarifying） |
| 7 | 需求与澄清一致 | 澄清结果已同步反映到需求描述区 |

3. 校验流程：
   - **不通过且不含待澄清项**：自动修订 spec，最多 **3 轮**
   - **不通过但含待澄清项**：跳过澄清相关校验项，其余继续
   - 3 轮后仍不通过的项标记为 `[REVIEW NEEDED]`

## 输出

- `specs/<feature>/spec.md`
- `specs/<feature>/checklists/requirements.md`
- 若生成的 `spec.md` 中包含待澄清问题（CQ），**MUST 明确高亮提示用户**：“发现需求盲区，请直接回复解答上述 Q&A 问题。解答完毕后，请按提示调用 `/clarify` 命令将您的回答澄清回填更新至文档。”
- 若已全部澄清回填完毕且用户确认无误，**MUST 明确提示用户**：“需求已收敛闭环，请调用 `/plan` 命令开始生成执行计划。”
