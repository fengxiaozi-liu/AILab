# 修复计划模板

在 `Plan` 阶段，按以下结构写入 `specs/<feature>/issue.md`。

```md
## 修复计划

**关联问题**:
- ISSUE-001

**修复目标**:
[本次修复要恢复的正确行为]

**根因分析**:
[说明历史 AI 为什么会形成该决策，以及该决策主要依据什么]
[明确归因为：SKILL、仓库事实，或二者共同作用]
[若历史决策存在偏差，继续说明偏差来源，例如：SKILL 约束、模板表达、仓库事实或二者映射关系]
[分析时优先沿 `spec.md -> plan.md -> tasks.md -> 实现` 的链路追溯决策来源]

**影响范围**:
- [受影响的模块、文件、流程]

**修复边界**:
- [允许修改的范围]
- [明确不做的范围]

**决策分析**:
[说明本次为什么选择当前修复方案，以及该方案主要依据什么]
[明确归因为：SKILL、仓库事实，或二者共同作用]
[若同时依赖二者，说明边界约束主要来自哪里，具体落点主要来自哪里]
[可补充未采用方案及其放弃原因，避免修复范围扩散]

**验证方式**:
- [测试、构建、日志、人工验证]
```

要求：

- `关联问题` 必须列出本次计划覆盖的全部未解决 `ISSUE-xxx`
- `根因分析` 必须回答“历史 AI 为什么这样决策”以及“该决策主要依据什么”
- `决策分析` 必须回答“本次为什么这样修”以及“该方案主要依据什么”
- `修复边界` 必须包含至少一条明确不做项，防止修复过程扩散
- `验证方式` 必须能直接对应后续 `Implement` 阶段的验收动作
