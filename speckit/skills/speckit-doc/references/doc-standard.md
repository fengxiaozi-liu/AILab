# Speckit Doc Standard

## 目标

`speckit-doc` 的目标不是生成“好看的说明文”，而是生成一份可持续维护、可追溯、可用于团队协作的标准化架构文档。

## 标准化原则

### 1. 事实优先

- 优先基于代码、配置、README、现有文档确认事实
- 若内容来自 `spec.md`、`plan.md`、`tasks.md`，必须区分：
  - 已实现事实
  - 目标设计
  - 待实现内容

### 2. 结构固定

`docs/architecture.md` 必须固定包含：

1. 文档概览
2. 总体架构说明
3. 总架构流程图
4. 核心功能与核心流程图
5. 实体设计
6. 关键约束与边界

### 3. 术语统一

- 同一模块、实体、流程名称在全文中必须保持一致
- 若项目已有命名方式，以项目现有命名为准
- 不得把某个具体项目的章节样式、示例名称或业务术语固化进通用标准

### 4. 图示统一

- 系统级结构优先使用 `graph TD`
- 核心交互流程优先使用 `sequenceDiagram`
- 图中节点名称要体现职责，不使用过度抽象的临时命名
- 图必须服务于理解，不追求完整穷举

### 5. 实体抽象适度

- “实体设计”面向业务理解和架构沟通
- 不等同于数据库 ERD，也不要求罗列全部字段
- 仅保留关键职责、关键属性和核心关系

### 6. 增量更新可追踪

- `update` 必须优先保留原文档稳定结构
- 只更新受当前 feature 影响的模块、流程、实体与约束
- 更新结果直接合并进对应章节，不追加变更历史区域

## 文风要求

- 使用陈述句，避免口号式描述
- 先讲边界和职责，再讲技术细节
- 每一节都要面向“新加入团队的工程师能快速建立系统认知”

## 输出要求

- 目标文件固定为 `docs/architecture.md`
- 文档中至少包含 1 个总架构 Mermaid 图
- 文档中至少包含 1 个核心流程 Mermaid 图
- 文档中必须存在核心实体表
- 若事实缺失，显式写出“待补充”，不留空白占位
