# Spec Kit Checklist Skill（中文）

## 何时使用

- **推荐时机**：clarify 完成、spec 变为 Ready 后，plan 之前——确认需求本身质量过关再进入技术设计
- 也可在流水线任意阶段使用（spec 存在即可）
- 检查的是“需求写得是否好”，而非验证代码实现是否正确

## 核心理念

- 清单是"需求的单元测试"
- 检查的是需求质量：完整性、清晰性、一致性、可测性、覆盖度
- **不是**验证实现行为（不能写"调用接口是否返回成功"这类条目）

## 输入

- 用户说明的清单焦点与范围
- 可用的 spec/plan/tasks（按需读取）

## 工作流

### §前置检查

1. 确认 `specs/<feature>/` 目录存在且至少有 `spec.md`。
2. 若 spec 不存在，终止并提示先运行 `/specify`。
3. 弃用独立清单文件方案。准确定位：checklist 的产出是用来“给 spec 挑刺”的，所有发现的漏点必须反向转化为 CQ 注入回 `spec.md`。

### §澄清问题

动态提问（最多 3 个，必要时再加 2 个）：

- 只问会实质改变清单内容的问题
- 可问的维度：范围、风险优先级、深度、受众、排除边界
- 已明确的信息不重复追问
- 若无法互动，默认：深度=Standard，受众=Author，焦点=相关性最高两类

### §主题确定

合并用户输入 + 澄清答案，确定清单主题。常见主题：

| 主题 | 适用场景 |
|------|---------|
| 功能完整性 | 检查 RQ 是否覆盖所有业务场景 |
| 接口设计 | 检查 Proto 接口定义的需求是否清晰 |
| 数据模型 | 检查实体/属性/关系的需求是否明确 |
| 安全 | 检查权限、认证、数据保护的需求是否完整 |
| 边界场景 | 检查异常流、限制条件是否有需求覆盖 |
| 非功能属性 | 检查性能、可用性等非功能需求是否量化 |

### §文档读取

按需读取 `specs/<feature>/` 下的文档：

| 文档 | 提取内容 |
|------|---------|
| spec.md | RQ 列表、关键实体、约束与边界 |
| plan.md（若存在） | 设计表、框架适配、风险项 |
| tasks.md（若存在） | 任务列表、Phase 覆盖 |

### §清单转化与生成

1. 按主题分类发掘潜在的业务盲区、边界遗漏或逻辑矛盾。
2. 将找出的缺漏点全部转化为**标准的 Q&A 待澄清问题（CQ）**，准备追加到 `spec.md` 现有的 `## 待澄清问题` 列表末尾。
3. 每条 CQ 必须是**填补需求质量漏洞**的问句（而不是测试实现行为的问句）。

#### 条目写作规则

**推荐维度**：
- 需求完整性（Completeness） / 边界覆盖（Edge Case Coverage）
- 需求清晰性（Clarity） / 可测性（Measurability）
- 需求一致性（Consistency） / 歧义与冲突（Ambiguities）
- 非功能需求（Non-Functional） / 依赖与假设（Dependencies）

**条目模式示例（必须严格遵循 Q&A 提问结构）**：

```markdown
### CQ-004: 错误处理与重试机制缺失

> **类别**: 边界覆盖（Edge Case Coverage） [Gap]

**Q (提问)**:
针对核心服务调用失败的场景，是否需要定义降级或重试机制？当前 spec 中未体现如果下游返回 503 时的默认行为。
*参考选项（如有）*：
- A. 需要重试 3 次后抛出异常
- B. 直接透传失败，要求人工介入

**A (澄清结论)**:
[待澄清]
```

**追踪性要求**：
- 在 `> **类别**` 或提问正文中，至少 50% 必须包含底层追溯标签：
  - 引用的需求块：`[Spec §RQ-xxx]`
  - 发现的缺陷类型：`[Gap]` / `[Ambiguity]` / `[Conflict]` / `[Assumption]`

**严禁内容**：
- 必须是讨论**业务需求缺口**！绝对禁止使用 "Verify/Test/Confirm/Check + 实现行为" 等 QA 测试人员执行系统测试的语句。
- 严禁要求验证具体的代码调用、执行算法、返回字段匹配等编码层面的细节。

### §写入规定（回写 Spec）

- **定向注入**：打开现有的 `specs/<feature>/spec.md`，找到底部的 `## 待澄清问题` 区域。
- **沿用序号**：读取当前最大的 CQ 序号（如 CQ-003），为你新发现的漏洞顺延编号（CQ-004, CQ-005...）。
- **强制 Q&A 格式**：使用标准的 `> 类别`、`**Q (提问)**`、`**A (澄清结论)**` 格式将其插入。

### §输出报告

完成后报告：
- 新增的 CQ 数量与涵盖的主题（安全、边界等）
- 深度级别与焦点主题

## 输出

- **目标更新文件**：`specs/<feature>/spec.md` (追加更新)
- **硬性提示要求**：完成后，**MUST 明确高亮提示用户**：“已通过深度 Checklist 审查发现了 N 个潜在的边界/完整性漏点，并已作为新的待澄清问题（CQ）追加至 `spec.md` 底部。请您前往解答，解答完毕后调用 `/clarify` 命令将变更正式落实到需求正文中。”
