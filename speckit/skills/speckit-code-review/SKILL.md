# Spec Kit Code Review Skill

## 何时使用

- implement 执行完毕后，对代码做质量审查
- 修复后再次审查，直到通过
- 提交前做风险检查、回归检查、完成度检查

## 输入

- 首选：`specs/<feature>/tasks.md`
- 首选：`specs/<feature>/plan.md`
- 可选：`specs/<feature>/spec.md`
- 实际代码文件
- 可选：测试文件、配置文件、接口定义、迁移文件等辅助上下文

## 目标

在代码提交前，识别完成度缺口、正确性问题、架构风险、性能隐患、安全问题和测试缺口。

## 职责边界

- `speckit-code-review` 负责通用代码审查流程和报告结构
- 当前项目是什么类型，由 agent 在运行时识别
- 需要加载什么项目适配 skill 或语言 skill，由 agent 在运行时决定
- 本 skill 不静态绑定任何具体项目技能或框架技能
- 如果项目类型无法识别，则退化为通用审查，并显式记录不确定性

## 约束

- 只读审查：输出报告，不自动修改代码
- 报告写入 `specs/<feature>/review.md`；如果没有 feature 上下文，则输出为一次当前任务对应的审查结果
- 每次审查生成完整报告，覆盖上一次结果

## 工作流

### 1. 审查模式判定

先判断是否存在 Speckit 上下文：

- 若存在 `specs/<feature>/tasks.md` 和 `plan.md`：进入 feature review 模式
- 若缺少其一或全部缺失：继续进入项目识别驱动审查

随后：

1. 识别当前项目类型
2. 判断是否需要加载项目适配 skill 或语言 skill
3. 若识别失败，则退化为通用 review，并记录不确定性

说明：

- `spec.md` 用于需求追溯；不存在时不阻塞审查
- 项目 skill 只能补充项目特有检查，不替代通用审查主干

### 2. 上下文加载

#### Feature Review 模式

从文档提取：

| 文档 | 提取内容 |
|------|---------|
| tasks.md | 已完成任务 `[x]`、文件路径、需求引用、Phase 分组 |
| plan.md | 设计约束、架构意图、测试策略 |
| spec.md | 需求列表、边界、约束（若存在） |

从代码提取：

- 读取 tasks.md 中已标记 `[x]` 的实际文件
- 同时读取与这些改动直接相关的测试和配置文件

#### 项目识别驱动审查

agent 需要结合仓库事实识别当前项目类型与结构。

通用项目类型示例：

- 服务端项目
- 库 / SDK 项目
- CLI 工具
- 前端应用
- 网关 / 接入层项目
- 数据处理 / 批处理项目

如果 agent 已识别到项目适配 skill 或语言 skill，则将这些 skill 的 MUST 规则作为审查补充输入。  
如果项目类型识别不确定，则继续按通用审查流程执行，并显式标记假设与风险。

#### 通用 Review 模式

当项目类型无法可靠识别时，从当前变更或目标文件中提取：

- 入口逻辑和核心实现
- 相关测试
- 关键配置、接口定义、数据结构、迁移或脚本

优先读取：

1. 核心业务逻辑
2. 数据访问或外部交互逻辑
3. 对外接口层
4. 测试与配置

### 3. 审查维度

按以下 6 个维度逐项审查。无论是否加载项目 skill，这些维度始终存在。

#### A. 完成度

| 检查项 | 规则 |
|--------|------|
| 文件存在 | 任务声明的文件或实现涉及的关键文件必须存在 |
| 方法实现 | 关键函数必须有实际实现，不能是空壳或 TODO |
| 需求覆盖 | 有 spec 时，每个关键需求至少有一个对应实现；无 spec 时，检查用户目标是否落地 |
| 变更闭环 | 代码、测试、配置、文档之间的必要联动是否完成 |

#### B. 正确性

| 检查项 | 规则 |
|--------|------|
| 逻辑正确 | 主流程、分支、边界条件和异常路径符合预期 |
| 错误处理 | 错误被正确返回、转换或处理，不吞错 |
| 返回语义 | 返回值语义清晰，不引入 `nil, nil` 或等价歧义 |
| 资源释放 | 文件、连接、Body、锁等资源被正确释放 |
| 并发安全 | goroutine 生命周期、共享状态、同步方式合理 |

#### C. 架构与可维护性

| 检查项 | 规则 |
|--------|------|
| 职责边界 | 逻辑放在合适层级，没有明显职责错位 |
| 依赖方向 | 依赖关系清晰，不引入反向依赖或隐式耦合 |
| 命名与表达 | 命名准确，代码意图清晰 |
| 抽象合理 | 没有过度抽象、重复抽象或临时性堆砌 |
| 复杂度扩散 | 不把不确定性和判断负担转嫁给调用方 |

#### D. 性能

| 检查项 | 模式 / 特征 |
|--------|------------|
| N+1 查询 | 循环内重复查询或重复外部调用 |
| 无界读取 | 缺少分页、限制或过滤 |
| 重复计算 | 循环中重复执行不变量计算或查询 |
| 不必要分配 | 大对象复制、频繁创建临时对象 |
| 阻塞风险 | 长时间锁持有、串行等待、不必要同步 |

#### E. 安全

| 检查项 | 模式 / 特征 |
|--------|------------|
| 注入风险 | SQL、命令、模板、脚本或其他拼接式注入 |
| 权限问题 | 缺少权限、租户、角色或所有权校验 |
| 敏感数据 | 密码、token、手机号、证件号等未脱敏 |
| 日志泄露 | 日志输出敏感信息 |
| 硬编码密钥 | secret、password、token 等硬编码在代码中 |
| 输入校验 | 外部输入缺少必要校验或约束 |

#### F. 测试

| 检查项 | 规则 |
|--------|------|
| 主路径覆盖 | 核心成功路径有测试覆盖 |
| 错误路径覆盖 | 关键失败路径、异常路径有测试覆盖 |
| 边界覆盖 | 边界输入、空值、极端值、非法值有测试覆盖 |
| 回归保护 | 测试能防止本次改动再次回归 |

### 4. 防御式编程补充检查

额外检查以下问题：

- 是否存在重复参数校验
- 是否存在 `nil, nil` 或等价歧义返回
- 是否存在吞错，包括 `_, _ = call()`、忽略 `err`、仅记录日志后继续伪装成功
- 是否把下层不确定性扩散成上层 `if obj != nil` 一类补判

默认严重级别：

- `nil, nil`、吞错：至少记为 `MAJOR`
- 重复参数校验、上层补判扩散：至少记为 `MINOR`

### 5. 严重级别

| 级别 | 定义 | 处理方式 |
|------|------|---------|
| `BLOCKER` | 功能未实现、核心需求未覆盖、严重安全问题 | 必须修复后才能提交 |
| `MAJOR` | 逻辑错误、架构风险、明显性能问题、吞错、歧义返回 | 强烈建议修复 |
| `MINOR` | 命名、结构、小型设计问题、重复校验等 | 建议修复 |
| `INFO` | 风格优化、可选改进项 | 可选处理 |

### 6. 报告生成

输出 Markdown 格式报告。

#### 报告结构

```markdown
# Code Review Report

> Feature: <feature-short-name or current-task>
> 审查时间: <timestamp>
> 审查模式: <feature-review / project-aware-review / general-review>

## 审查摘要

| 维度 | BLOCKER | MAJOR | MINOR | INFO | 评分 |
|------|---------|-------|-------|------|------|
| 完成度 | | | | | /10 |
| 正确性 | | | | | /10 |
| 架构与可维护性 | | | | | /10 |
| 性能 | | | | | /10 |
| 安全 | | | | | /10 |
| 测试 | | | | | /10 |
| **总计** | | | | | **/60** |

## 完成度

### 已完成内容

| Task ID / Area | 对应需求 | 描述 | 文件 | 状态 |
|----------------|---------|------|------|------|

### 未完成内容

| Task ID / Area | 对应需求 | 描述 | 影响 |
|----------------|---------|------|------|

## 发现清单

| ID | 维度 | 严重级别 | 文件:行号 | 摘要 | 建议 |
|----|------|---------|-----------|------|------|

## 修复建议优先级

按 `BLOCKER -> MAJOR -> MINOR` 排序，给出具体修复建议。
```

#### 评分规则

每个维度 10 分起，按发现项扣分：

| 级别 | 每项扣分 |
|------|---------|
| `BLOCKER` | -3 |
| `MAJOR` | -2 |
| `MINOR` | -1 |
| `INFO` | 0 |

最低 0 分，不出现负分。

### 7. 下一步建议

- 有 `BLOCKER`：必须修复后再审查
- 仅有 `MAJOR`：建议修复后再次审查
- 只有 `MINOR/INFO`：可以提交，但应列出后续改进建议
- 无 `BLOCKER/MAJOR` 且总分达标：审查通过

## 输出

- 审查报告：`specs/<feature>/review.md` 或当前任务对应的审查结果
- 完成后提示用户 review 审查报告，确认所有问题已修复后继续执行 `/summary` 生成完工总结。
