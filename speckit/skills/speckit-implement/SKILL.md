# Spec Kit Implement Skill

## 何时使用

- 已存在 `specs/<feature>/tasks.md`，需要进入正式实施阶段
- 用户明确要求按任务清单逐项落地实现

## 核心原则

以下规则均为强制规则，不可弱化，不可绕过：

1. 必须存在 `tasks.md`
   未发现 `specs/<feature>/tasks.md` 时，不得进入实现阶段，不得自行跳过，也不得用口头计划替代。
2. `tasks.md` 是唯一执行清单
   实施范围、断点续做、交付边界，均以 `tasks.md` 为准。
3. Phase 内连贯执行
   允许在一个 Phase 内部，将关联度高、涉及相同模块或具有上下文连续性的多个任务项合并实施，但在提交代码前必须保持逻辑清晰。
4. Phase 内的周期性勾选与绝不拖延原则
   不强求“每做完一个小任务就立刻去更新一次 `tasks.md`”。允许在完成一个连贯小代码批次后，统一把这几个小任务一起从 `[ ]` 更新为 `[x]`。**但极其严禁“把打钩动作当成了拖延症”，绝不允许等到所有的代码任务全敲完之后才想起来统一回填。**
5. 严禁跨 Phase 并行与越界（红线）
   必须确保在进入下一个 Phase 之前，**当前 Phase 下的所有任务都已经在代码层面实现了落地，并且在 `tasks.md` 里物理打上了 `[x]` 勾选印记**。一旦发现在当前 Phase 未打完钩时就抢跑去写下一个 Phase 的代码，或者企图把所有 Phase 攒到最后一起大批量回填打钩，均视为严重违章并立刻终止。
6. 前瞻设计与本分实现
   允许且鼓励前瞻性阅读当前 Phase 及后续 Phase 的任务，以确保底层类、接口的架构合理性。但在具体编写代码时，仅修改当前所处任务需要变更的实体逻辑，不抢跑细枝末节。
7. 禁止擅自改写任务边界
   不得在未说明原因的情况下随意合并任务、拆分任务、重排 Phase 顺序或修改验收标准。
8. 命中相关任务必须加载对应 skill
   如果当前 Phase 内的任务明显对应某个已安装 skill 的适用范围，必须先加载该 skill，再按该 skill 的规则完成该批次任务。

## 输入

- 必需：用户目标、范围、约束、禁止项、验收口径
- 必需：`specs/<feature>/tasks.md`
- 可选：`specs/<feature>/plan.md`
- 可选：`specs/<feature>/spec.md`
- 可选：仓库内与当前任务直接相关的代码、测试、脚本、配置文件

## 职责边界

`speckit-implement` 只负责“基于既有任务清单实施并闭环”，不负责在没有 `tasks.md` 的情况下直接开工。

- 如果没有 `tasks.md`：停止实施，并明确指出缺少前置交付物
- 如果 `tasks.md` 存在但内容明显不完整：先向用户报告缺口，不得擅自越过缺口继续大面积编码
- 如果需要补任务清单：应引导使用 `speckit-tasks`，而不是在本技能中直接替代

## Skill 加载与纠偏机制

执行每一个 Phase 前，必须统一扫描该 Phase 内的所有任务是否触发其他相关 skill。由于在执行中可能会出现误判，在此设定**去重**与**漏检兜底**机制。

强制规则如下：

1. **Phase 级扫描与去重加载**
   - 在 Phase 开始时统一扫描本阶段所有任务。
   - **绝对去重**：如果该 Phase 内有多个任务均涉及同一个 Skill，**只允许在 Phase 开启时加载且声明一次**。严禁在实施每一个微小子任务前都像复读机一样重复宣告加载该 Skill。
   - **跨 Phase 缓存**：如果上一 Phase 已经加载并吃透了某个全局型 Skill（如纯粹的代码风格审查规范），且本 Phase 继续沿用，无需再次发起物理文件的重新宣读动作。
2. **中途漏检的动态纠偏补载**
   - **发现就停**：如果在敲代码的中途（或是在节点自检时），系统突然发觉涉及到了高度专业的领域（如忽然开始手写复杂 SQL、突然碰到未曾预见的框架网关配置等），而此时并未加载对应的专职 Skill，必须**立刻原地停止瞎编**。
   - **动态挂载重建**：立刻打断当前步骤补加载缺失的 Skill，并根据新载入的 `MUST` 规则对刚才凭直觉写的代码进行全量二次 review 和重构。
3. **命中即加载**
   只要任务描述、目标文件、实现类型、验收动作明显落入某个 skill 的适用范围，就必须预先加载该 skill。
4. **MUST 规则自动继承**
   一旦加载，被加载的 skill 中的 `MUST / 强制规则` 视为当前的顶级附加约束，绝对不得降级执行。
5. **多 skill 无缝叠加与冲突暴露**
   如果同一批次任务同时命中多个 skill，必须尽数遵守并综合考量；若存在无法弥合的规则冲突，必须显式报告给用户并停止推进。
6. **通用流程兜底**
   只有在当前所有任务均未命中额外专属 skill 时，才按本技能最初的通用编码规则孤独推进。

## 工作流

### 1. 前置检查

1. 确认 `specs/<feature>/tasks.md` 存在
2. 读取 `tasks.md`，识别：
   - Phase 划分
   - 任务 ID
   - 任务描述
   - 目标文件或模块
   - 完成状态 `[ ] / [x]`
3. 如存在 `plan.md` 或 `spec.md`，读取与当前直接相关的边界、设计约束、验收标准
4. 识别当前项目类型与适配规则来源

若步骤 1 失败：立即停止，不进入实现。

### 2. 建立 Phase 级上下文

从 `tasks.md` 中定位“第一个未完全闭环的 Phase”作为当前焦点。

执行游标规则：

- 一次聚焦一个 Phase，可以提取本 Phase 所有 `[ ]` 任务进入上下文。
- 在当前 Phase 闭环前，下一 Phase 的具体实现视为锁定状态（仅许进行架构前读预判）。
- 已完成 `[x]` 的任务只用作上下文参考，不重复实施。

### 3. 本地连贯执行循环

对焦点 Phase 内的任务，按以下顺序执行：

1. **预研与加载**：扫描 Phase 内任务描述，按需加载对应 skill 及周边依赖文件。
2. **综合实现**：结合 `plan.md` / `spec.md` 形成代码。允许把同一模块强相关的多个任务并作一次代码修改。
3. **节点验证（可选）**：鼓励在大幅代码更改后，调用相关的本地验证工具（如 Lint检查，类型编译构建，或跑某个单元测试）以保证质量。
4. **统一打钩**：当前批次（或该 Phase）代码落地与自检通过后，一次性将涉及的任务记录为 `[x]`。 
5. 定位下一批次未完成任务或下一 Phase。

### 4. 强制门禁与条件降级

有阻断性冲突时，停在当前状态：

- 缺乏必要上下文，无法安全实施
- 涉及严重的设计冲突，无法自行裁决
- 依赖的生成步骤或核心技能调用彻底失败
- 需要修改 `tasks.md` 整体结构且无用户确认

**局部非阻断失败（降级处理）**：

- 若某个子项非致命性失败（例如某个生僻测试库报错），此时不强求全盘卡死。可以将其状态单独注记为 `[blocked]` 面向用户说明，并优先清理本 Phase 内的其他独立任务，最后统一寻求用户确认。

停机或等待协助时必须输出：

- 卡住的具体任务项
- 阻塞原因及其相关报错日志
- 本阶段已完成的成果
- 期望用户协助的决断项

### 5. 关于 Phase 的约束

Phase 是任务的主要交付边界。

- 必须按 Phase 汇报整体进度
- **坚决禁止**跨 Phase 批量开发后再统一勾选
- 一个 Phase 完全打上勾 `[x]` 后，才能正式对外宣告该阶段结束并顺延到下一阶段

### 6. 关于 `tasks.md` 的更新规则

`tasks.md` 的状态更新必须满足以下要求：

- **允许小批次更新**：不需要做一条打一个勾；但在上下文切换或阶段落幕前，必须物理写入文件。
- **状态不造假**：必须在对应的终端命令行或文件系统证实代码已写入后，方能标记 `[x]`。

### 7. 断点续做

如果 `tasks.md` 中已有若干部分标记 `[x]`：

1. 寻找第一个包含 `[ ]` 的 Phase
2. 以该 Phase 为上下文起点，略过其中已完成项，专门针对剩余 `[ ]` 项执行连贯工作流
3. 如果中途人工改动了打钩，需静默重新加载并校准游标

### 8. 完成校验

只有当 `tasks.md` 中所有任务项都已按规则实施后，本实施阶段才算结束。

最终必须同时满足：

| # | 校验项 | 标准 |
|---|---|---|
| 1 | 任务闭环 | `tasks.md` 中所有主体任务均为 `[x]` |
| 2 | Phase 边界 | 未出现跨大阶段杂糅的提交和勾选行为 |
| 3 | 需求覆盖 | 若 `spec.md` 存在，每条需求均按序落下 |
| 4 | 设计一致 | 若 `plan.md` 存在，实现与设计约束一致 |
| 5 | Skill 合规 | 命中的相关 skill 均已按 Phase 加载并遵守 MUST 规则 |

## 实施输出

技能执行完成后，应输出：

- 代码实现改动
- 更新后的 `specs/<feature>/tasks.md`
- 每个命中的相关 skill 及其使用说明
- 如全部完成，提示进入 `/code-review`

## 执行时的明确表述要求

在实施过程中，agent 的表达应体现出“全局考量，批次落盘”，例如：

- “当前所处阶段：Phase 1，正处理该阶段内部的任务 [T1, T2]”
- “鉴于本 Phase 针对 UI 开发，已预加载 `vue-skill` 的强约束”
- “T1, T2 模块耦合度强，现统一修改，并执行检查”
- “本批次代码已落盘，正在更新 tasks.md，将 [T1, T2] 并行勾选为 `[x]`”

禁止出现的执行方式：

- “我先把所有 Phase 的任务全写完，最后发一张总表” (跨界大锅烩)
- “T3 失败卡死了，本 Phase 其他独立的模块也不搞了” (过激卡死)
- “这个任务明显需要相关 skill，但先凑合改代码回头再补” (跳过合规)

## 强制输出

阶段开启前，剖析该阶段内的任务需要绑定哪些独立的 skill，并以 Markdown 语块形式强制输出下方结构宣告：

**🔴 阶段执行开启 🔴**
- **Current Phase**: Phase 1: xxx
- **Skills Loaded**:
  - `T001` 命中 `vue-skill`, `ai-skill`
  - `T002` 命中 `ai-skill`

阶段结束，完成批量勾选 `[x]` 并确认落盘后，以明确的标识输出下方宣告结构（带上完成清单以便对账）：

**🟢 阶段执行闭环 🟢**
- **Completed Phase**: Phase 1: xxx
- **Tasks Done**: `T001`, `T002`, `T003`
- **Writeback Status**: Success
- **Next Phase**: Phase 2: yyy
