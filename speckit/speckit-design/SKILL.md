# Spec Kit Design Skill（中文）

## 何时使用

- 需求复杂，存在多个可行方案，需要做取舍与风险评估。
- 用户提出“方案设计/架构设计/技术选型/要不要做成事件驱动/如何分层/如何落库/如何拆服务”等问题。
- 需要输出可落地的设计文档，并能映射到后续 plan/tasks。

## 输入

- 用户需求描述（自然语言或已有 spec）
- 可选：`specs/<feature>/spec.md`
- 仓库现状（目录结构、现有模块、已有依赖/中间件、约束规则）

## 依赖 Skill（强制联动）

本技能是“框架无关”的方案设计能力，不强绑任何框架。

当项目属于某个框架/平台时，必须加载对应“框架适配器技能”，保证方案可落地且符合工程规范：

> 规则：先适配器（框架规范），后方案（业务/技术权衡），最后落地（plan/tasks/实现）。

## 工作流

### §0 证据与约束收集（禁止臆测）

1. 读取项目宪法与规则（如存在）。
2. 用工具确认仓库事实：相关目录/文件、已有接口、已有模型、已有依赖与约束。
3. 明确非功能目标：性能、可靠性、可观测性、安全、发布/回滚。

输出（简短）：
- `Constraints:` 约束清单
- `Assumptions:` 假设清单（可被验证/可被反驳）
- `OpenQuestions:` 需要用户确认的问题

### §1 三方内部讨论（单轮可完成，必要时两轮）

以三种角色分别写观点，每个角色必须包含“主张 + 反对点 + 需要的证据/数据”。

#### 角色 A：高级产品（Product）

- 目标用户/核心场景
- 成功指标与验收口径
- 业务边界与非目标（明确不做什么）
- 风险与灰度策略（对业务影响）

#### 角色 B：资深架构师（Architect）

- 方案分解（分层/模块/边界）
- 数据与一致性策略（事务、幂等、补偿）
- 依赖与集成策略（内外部系统、调用方式）
- 演进与可维护性（扩展点、兼容性、迁移）

#### 角色 C：资深开发（Senior Dev）

- 落地复杂度与实现路径（按 Kratos 结构落位）
- 性能/可靠性风险点与压测点
- 可测试性与可观测性设计
- 典型坑与回归风险（需要的测试/监控）

### §2 方案枚举与权衡（至少 2 个方案）

输出至少两套候选方案（A/B；复杂时可 A/B/C），每套必须包含：

- `Architecture:` 组件与边界（文字即可）
- `Data:` 表/模型/索引/关系与迁移策略（如涉及）
- `API:` proto/HTTP/gRPC 变化（如涉及）
- `Flow:` 关键链路时序（文字步骤）
- `Risks:` 风险与缓解
- `Cost:` 工期与复杂度（粗粒度）
- `Fit:` 与项目规范/技能的一致性自检（点名技能）

### §3 决策记录（Decision）

必须给出明确结论：

- `Decision:` 选哪个方案，为什么
- `Rejected:` 拒绝其他方案的主要原因
- `Follow-ups:` 后续必做事项（测试、监控、迁移、灰度、文档）

### §4 输出落盘（可选但推荐）

默认输出一个设计文档（若仓库有 specs 结构则写入，否则只在对话输出）：

- `specs/<feature>/design.md`：方案设计与决策记录

并建议把关键结论同步到：

- `specs/<feature>/plan.md`（若存在）
- `specs/<feature>/tasks.md`（若存在）

## 输出格式（强制）

最终输出必须包含这些一级标题（顺序固定）：

1. `背景与目标`
2. `约束与边界`
3. `方案候选`
4. `权衡与决策`
5. `落地与验证`
6. `风险与回滚`

并且必须包含一段“技能点名清单”：

- `SubSkills:` 本次方案涉及并应加载的 skills/patterns 列表（包含框架适配器，如 `kratos-patterns`，以及语言/工程模式技能）

## 质量门禁（自检）

- 是否至少 2 个候选方案？
- 是否给出明确决策与拒绝理由？
- 是否点名了需要加载的技能/适配器（而不是指向某个文件路径）？
- 是否包含可执行的落地与验证策略（测试/生成/构建/观测）？
