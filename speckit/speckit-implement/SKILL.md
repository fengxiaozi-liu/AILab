# Spec Kit Implement Skill

## 何时使用

- 已有 `tasks.md`，准备按任务清单正式开发
- 没有 `tasks.md`，但用户希望直接实现；此时先生成最小可执行任务清单草案，再继续实施

## 输入

- 必需：用户输入，包括目标、范围、约束、可跳过项、验收口径
- 可选：`specs/<feature>/tasks.md`
- 可选：`specs/<feature>/plan.md`
- 可选：`specs/<feature>/spec.md`
- 用户约束，例如范围、跳过的 Phase 等

## 职责边界

- `speckit-implement` 负责按任务执行、阶段推进、校验闭环
- 当前项目是什么类型，由 implement agent 在运行时识别
- 需要加载什么项目适配 skill 或语言 skill，由 agent 在运行时决定
- 本 skill 不静态绑定任何具体项目技能或框架技能

## 工作流

### 1. 前置检查

1. 若存在 `specs/<feature>/tasks.md`：进入按 tasks 执行模式
2. 若 `tasks.md` 不存在：基于用户输入生成最小任务清单草案，再进入实施
3. 若存在 `specs/<feature>/plan.md`：读取设计要点和执行约束；若不存在，则结合仓库事实与用户输入继续实施

### 2. 上下文加载

1. 若存在 `tasks.md`：解析 Phase 列表、任务 ID、描述、文件路径、完成标记，用于断点续做
2. 若存在 `plan.md` 或 `spec.md`：读取设计边界、验收口径与关键约束
3. 识别当前项目类型与所需规则来源

通用项目类型示例：

- 服务端项目
- 库 / SDK 项目
- CLI 工具
- 前端应用
- 网关 / 接入层项目
- 数据处理 / 批处理项目

如果 agent 已识别到项目适配 skill 或语言 skill，则把这些 skill 的 MUST 规则作为当前实施约束。  
如果项目类型识别不确定，则继续按通用实施流程推进，并显式记录假设和风险。

### 3. 按 Phase 执行

按 Phase 逐步推进，每个 Phase 完成后暂停汇报，等待用户确认再继续。

每个 Phase 内：

1. 识别本 Phase 需要遵循的规则来源
2. 根据 `plan.md`、`tasks.md`、仓库事实和当前适配约束编写代码
3. 按顺序完成任务，并在 `tasks.md` 中标记 `[x]`
4. 汇报本 Phase 完成内容、产出文件和下一阶段计划

### 4. 适配与实施原则

- 如果 agent 已识别到项目适配 skill：按该 skill 的 MUST 规则约束实现
- 如果 agent 已识别到语言 skill：按语言层面的实现规则约束实现
- 如果两者都未识别到：按 `plan.md`、`tasks.md` 与仓库事实执行通用实现
- 如果适配约束之间存在冲突：优先保持用户目标、仓库事实和当前任务边界一致，并显式记录冲突点

### 5. 生成与验证

如果某个 Phase 涉及代码生成、构建或其他自动化产物：

- 执行后必须验证生成结果无报错
- 若生成失败，停止并报告错误详情
- 生成成功后再继续后续 Phase

### 6. 断点续做

如果 `tasks.md` 中已有 `[x]`：

1. 跳过已完成任务
2. 从第一个未完成任务所在的 Phase 开始
3. 汇报续做起点

### 7. 错误处理

- 任务执行失败：立即停止，报告错误详情，等待用户决策
- 不跳过失败任务继续后续实施
- 如果项目类型识别不确定导致风险较高：显式报告该不确定性，而不是假装已确认

### 8. 完成校验

所有 Phase 执行完毕后：

| # | 校验项 | 标准 |
|---|--------|------|
| 1 | 任务完成 | `tasks.md` 中所有任务均已完成 |
| 2 | 需求覆盖 | 若 spec 存在，每条需求均有对应实现；若 spec 不存在，每条临时需求点均有对应落地或说明 |
| 3 | 设计一致 | 若 plan 存在，代码与 `plan.md` 一致；若 plan 不存在，代码与用户确认的最小任务清单一致 |
| 4 | 构建通过 | 代码可编译、构建或通过必要验证 |
| 5 | 适配可追溯 | 若使用了项目适配 skill 或语言 skill，其约束来源可说明 |
| 6 | 风险显式 | 若项目识别存在不确定性，相关假设与风险已说明 |

## 输出

- 代码实现改动
- 更新后的 `specs/<feature>/tasks.md`
