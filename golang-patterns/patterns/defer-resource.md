# Defer Resource Pattern

## 这个主题解决什么问题

说明资源释放、锁释放和状态恢复时如何使用 `defer`。

## 适用场景

- 文件、连接、事务、锁
- 临时状态恢复

## 设计意图

资源释放模式的重点是让清理动作和资源获取位置保持稳定对应，减少隐藏泄漏和异常路径遗漏。

- `defer` 用得好时，主流程和清理流程可以同时保持清晰。
- 理解资源生命周期后，更容易在获取资源后立即安排收尾逻辑。
- 把收尾动作放在接近获取点的位置，比在函数尾部集中回收更易维护。

## 实施提示

- 先标出资源获取点，再就近安排释放逻辑。
- 如果清理动作也可能返回错误，提前决定日志记录还是合并主错误。
- 当函数存在多重返回路径时，优先选能覆盖全部路径的释放组织方式。

## 推荐模式

- 获取资源后第一时间 `defer` 释放
- 对称的状态修改用 `defer` 恢复

## 标准模板

```go
mu.Lock()
defer mu.Unlock()

f, err := os.Open(name)
if err != nil {
    return err
}
defer f.Close()
```

## 常见坑

- 多分支返回遗漏释放
- 高热循环里把 `defer` 用成性能瓶颈
