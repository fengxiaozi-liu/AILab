# Pkg Subdomain Spec

本规范用于指导在 `internal/pkg` 下新增子域（新目录）或新增对外导出 API 的最小落地方式。目标是让公共能力边界清晰、可复用、可维护，避免演化成“公共垃圾堆”。

## 何时新增子域

- 现有子域（例如 `util`/`metadata`/`middleware`）无法准确表达能力边界。
- 能力脱离具体业务语义后仍成立，并预计会被多个模块稳定复用。
- 能力需要一组稳定的对外导出 API（不是单个一次性函数）。

## 何时不要新增子域

- 只有一个调用点成立，或只为当前需求“看起来通用”。
- 能力依赖业务状态判断、Repo 访问、领域对象或外部业务语义。
- 能力本质是“组件接入/生命周期注册/consumer/listener/event”的实现细节（应留在对应层）。

## 命名与边界

- 子域目录名必须直接表达能力边界，避免 `common`、`helper`、`misc`、`base`。
- 一个子域只承载一类稳定能力面，避免跨主题混放。
- 子域内文件按主题拆分，文件名表达能力点，例如 `url.go`、`retry.go`、`language.go`。

## 最小创建流程

1. 先检索 `internal/pkg` 是否已有相同或相邻能力；若已有，优先扩展现有子域而不是新建目录。
2. 定义子域边界：写清楚“解决什么问题”和“明确不解决什么问题”。
3. 创建最小导出 API：
   - 优先 small surface area：少量函数/类型，避免预先设计抽象层。
   - 每个导出 API 必须能独立解释其稳定性与复用价值。
4. 选一个真实调用点接入新 API，并确保不引入业务语义依赖。
5. 回收检查：路径是否表达边界、是否重复造轮子、是否存在过度抽象。

## 导出 API 约束

- 导出 API 必须是稳定能力，不得把业务流程编排包装成“公共函数”。
- 不导出与具体业务强绑定的类型；必要时用基础类型或小型接口隔离。
- 避免引入大量可选参数与复杂配置；配置复杂度高时，考虑拆成更小的能力或保留在原层。
