# Skill Design Guide

## 目的

这份文档用于回答 4 个问题：

- 什么时候适合建立一个技能
- 一个好的技能应该是什么样子
- `references/` 适合放什么
- `assets/` 和 `scripts/` 分别适合放什么

本文偏工程实践，不是技能语法说明。重点是帮助团队判断“这个知识到底该不该做成 skill”，以及“做出来的 skill 是否真的有用”。

## 什么情况下适合建立一个技能

适合建立技能的前提，不是“这个领域很大”，而是“这个领域存在稳定、重复、非显然的执行规律”。

### 适合建立技能的典型场景

1. 同类任务会反复出现。
例如经常要判断代码落层、补 codegen、统一错误语义、接入组件、沉淀公共能力。

2. 任务里有项目私有知识。
例如目录边界、命名约定、联动链路、团队规范、仓库内特殊流程。

3. 仅靠模型常识，容易做出“看起来合理、实际上不符合项目约定”的决定。
例如把业务逻辑写到 `service`，或把单点 helper 下沉到 `internal/pkg`。

4. 任务可以拆出稳定的执行流程。
例如：
   先判定任务类型
   再加载对应 references
   再检查已有实现
   最后做验证

5. 任务有重复可复用的资源。
例如固定脚本、参考规范、模板、样例、静态资产。

### 不适合建立技能的场景

1. 只是通用知识，没有项目特有约束。
例如“怎么写 Go”“什么是 HTTP”“怎么写 SQL”。

2. 任务极少发生，且没有复用价值。

3. 没有稳定边界，什么都想管。
这种内容更像一份大而空的说明书，不像技能。

4. 只是把 README 换个名字塞进 skill。
如果内容不能直接提升 agent 的判断和执行质量，就不值得建 skill。

## 如何判断“该不该建”

可以用下面这个四问法：

1. 这个任务是否会反复出现？
2. 这个任务是否依赖项目私有知识？
3. 这个任务是否经常被做错？
4. 这个任务是否能沉淀成稳定流程、脚本、参考资料或模板？

如果 4 个问题里至少有 3 个答案是“是”，通常就适合做 skill。

## 一个好的技能应该是什么样子

一个好的技能，本质上是“给另一个 agent 的高质量上岗手册”。

### 好技能的核心特征

1. 触发边界清晰。
agent 能看懂什么时候该用它，什么时候不该用它。

2. Frontmatter 简洁且高命中。
`name` 和 `description` 是触发入口，必须优先写“能力”和“触发词”，不要写长篇解释。

3. 正文只保留执行所需的最小流程。
SKILL.md 不应该像百科全书，应该像作战卡片。

4. 能指导决策，而不只是堆概念。
好的技能会告诉 agent：
先看什么
怎么判断
何时切换方向
哪些坑不能踩

5. 能按需加载，而不是一次塞满上下文。
共性流程写在 `SKILL.md`，细节知识拆进 `references/`。

6. 有明确排他边界。
要写清楚“不处理什么”，避免和其他 skill 抢任务。

7. 有稳定的资源组织方式。
脚本进 `scripts/`
文档进 `references/`
输出素材进 `assets/`

### 一个坏技能常见的样子

1. 描述很长，但触发词很弱。
2. 边界含糊，和别的 skill 大量重叠。
3. 正文充满背景介绍，却缺少执行指令。
4. `references/` 和 `SKILL.md` 大量重复。
5. 什么都想指导，最后什么也指导不清。

## SKILL.md 应该承载什么

`SKILL.md` 应该只承载 4 类内容：

1. 这个技能是做什么的
2. 什么时候使用
3. 处理任务的最小工作流
4. 关键约束和 references 导航

推荐结构：

```md
---
name: xxx
description: 能力 + 触发词 + 排他边界
---

# Skill Name

## 何时使用

## 职责边界

## 输入

## 工作流

## 约束

## 参考文件
```

## references/ 适合放什么

`references/` 放的是“需要被读，但不该常驻在 SKILL.md 里的知识”。

### 适合放进 references/ 的内容

1. 详细规范。
例如错误语义规范、命名规范、Repo 边界规范、codegen 规则。

2. 项目私有知识。
例如目录结构说明、联动约束、特定模块说明。

3. 易变但需要查询的知识。
例如 API 约定、字段说明、生成链路、特定流程说明。

4. 只在某些子场景才需要的细节。
例如 `consumer-spec.md`、`wire-spec.md`、`comment-spec.md`。

5. 较长的示例和反例。

### 不适合放进 references/ 的内容

1. SKILL.md 已经明确写过的核心流程。
2. 完全通用、模型本来就知道的常识。
3. 与技能无关的背景介绍。
4. 大量重复、拷贝式资料。

### references/ 的写法建议

1. 一份 reference 只解决一个主题。
2. 文件名要直接表达主题，例如：
   `error-spec.md`
   `wire-spec.md`
   `consumer-spec.md`
3. 开头最好写适用场景。
4. 长文档最好有目录。
5. 避免 references 互相层层跳转太深。

## scripts/ 适合放什么

`scripts/` 放的是“重复出现、需要稳定执行、用脚本比让模型临时写更可靠”的东西。

### 适合放进 scripts/ 的内容

1. 重复性的机械流程。
例如代码生成、格式转换、文件提取、批量校验。

2. 容易写错但步骤固定的操作。
例如规范检查、结构转换、特定产物生成。

3. 每次临时写都浪费上下文的代码。

4. 需要高确定性的执行逻辑。

### 不适合放进 scripts/ 的内容

1. 只会执行一次的临时代码。
2. 高度依赖当前上下文判断的复杂策略。
3. 本质上是文档说明的内容。

### scripts/ 的写法建议

1. 让脚本职责单一。
2. 参数尽量清晰。
3. 能复用就不要把流程重新写回 SKILL.md。
4. 脚本需要真实可运行，而不是占位。
5. 如果脚本是关键能力，应在 SKILL.md 明确何时调用。

## assets/ 适合放什么

`assets/` 放的是“最终产出会使用，但不需要被大量读入上下文”的资源。

### 适合放进 assets/ 的内容

1. 模板文件。
例如前端模板、文档模板、配置模板。

2. 品牌与视觉资源。
例如 logo、字体、图标。

3. 样板工程或静态骨架。
例如初始化项目骨架、示例目录结构。

4. 固定素材。
例如示例文档、占位文件、可复制资源。

### 不适合放进 assets/ 的内容

1. 规则说明文档。
那应该进 `references/`。

2. 可执行逻辑。
那应该进 `scripts/`。

3. 仅供解释而不会被输出使用的内容。

## 如何在三类资源之间做选择

可以用这个判断：

- 主要用于“指导判断” -> `references/`
- 主要用于“稳定执行” -> `scripts/`
- 主要用于“参与输出” -> `assets/`

如果一份内容既要指导判断又要稳定执行，优先拆成两部分：

- 规则写进 `references/`
- 执行逻辑写进 `scripts/`

## 一个团队内好用的技能体系应该是什么样

对于复杂项目，通常不是“一个大技能”，而是“一个技能族”。

例如 Kratos 这种项目，更适合拆成：

- 判定型技能：判断项目类型、代码落层
- 主执行技能：处理 domain、service、components、pkg 这类主任务
- 审查型技能：做 style、review、收尾治理
- 约定型技能：管理共享语义、字典、注释、日志规范

关键不是拆得多，而是每个 skill 的职责要单一、边界要稳定。

## 技能设计的常见错误

1. 把 skill 写成大而全说明书。
2. 把多个高频任务硬塞进同一个 skill。
3. 只写“能做什么”，不写“何时不用”。
4. 把详细知识都堆进 SKILL.md，导致上下文膨胀。
5. 缺少可复用脚本，导致 agent 每次都重写同样逻辑。
6. 引用了很多 resources，但没有在 SKILL.md 里写清楚何时加载。
7. skill 名字和 description 不利于触发。

## 最小落地建议

如果要从零开始做一个技能，建议最少做到：

1. 写清楚 `name` 和 `description`
2. 写清楚“何时使用”和“职责边界”
3. 把执行流程压缩成 3 到 6 步
4. 把细节拆进 `references/`
5. 只有真正需要时才加 `scripts/` 和 `assets/`

## 对 Kratos 这类项目的建议

Kratos 这类项目适合建立技能，不是因为框架本身复杂，而是因为它有大量项目私有决策：

- 代码落层
- 边界划分
- 联动链路
- 约定式命名
- 共享语义
- 公共能力沉淀
- 收尾审查

这类知识非常适合以 skill 形式组织。

更准确地说，它适合建立“技能族”，而不是只做一个大 skill。
