# Aggregate Root

## 作用范围

本文用于说明在 Kratos 项目中如何识别稳定业务对象、确认聚合根，以及如何围绕聚合根组织领域知识。

当问题属于以下场景时，应优先查看本文：
- 判断当前业务应围绕哪个对象组织
- 判断某个对象、文件、接口或领域能力归属哪个聚合根
- 判断一个结构是聚合根、从属实体、关系视图还是临时投影
- 判断 `repo`、`usecase`、领域对象、应用层对象、`proto` 是否围绕同一领域中心收敛

本文重点回答聚合根识别与围绕谁组织领域的问题，不展开协议层细节、组件接线细节与公共能力下沉细节。

## 聚合根定位

聚合根是领域组织中心，不只是一个结构体名字，也不是接口动作、页面结构或临时返回体的别名。

聚合根的作用在于：
- 收口稳定业务边界
- 承载领域主语
- 为 `repo`、`usecase`、领域对象、应用层对象、`proto` 提供统一命名中心
- 让跨层对象在同一业务语义下保持收敛

如果对象名称无法稳定映射回某个聚合根，通常说明建模还没有收敛。

## 识别原则

识别聚合根时，应优先遵守以下原则：
- 先识别稳定业务对象，再确定聚合根
- 聚合边界先于接口动作、页面结构和临时返回体
- 不先按页面、按钮动作、临时查询结果切领域模型
- `repo`、`usecase`、`proto`、`service` 的命名与职责都应回收到同一聚合根

聚合根识别是领域建模问题，不是命名美化问题。

## 聚合根确认步骤

确认聚合根时，不只判断名字是否像主对象，还应显式确认该聚合根实体及其从属实体承载哪些稳定字段。

推荐按以下顺序判断：
1. 先确认哪个实体承载稳定业务主语，并以它作为聚合根候选。
2. 再确认聚合根实体与从属实体分别有哪些核心字段、状态字段、类型字段、结果字段、原因字段和阶段字段。
3. 判断这些字段中哪些属于有限值集合，哪些只是原始透传值、临时展示值或外部原始 payload。
4. 只要字段具备稳定值域并会被业务判断消费，就应建立对应的 enum 或 typed const，而不是长期保持裸 `string` / `int`。
5. 聚合根识别完成后，围绕该聚合根收敛 `repo`、`usecase`、领域对象、应用层对象、`proto` 以及相关 enum。

### 实体字段确认约束

聚合根及其从属实体字段确认时，应补做一轮“可枚举字段检查”：
- 如果字段值来自有限集合，例如状态、类型、结果、原因、阶段、动作，应优先定义为 enum 或 typed const
- 如果字段会参与 `switch`、分支判断、状态流转、结果映射或跨层重复消费，应视为稳定值域并收敛为 enum
- 如果字段只是第三方原始 payload、日志透传值或暂存原始响应，可以保留原始类型，但不应直接承载领域判断

这一步属于聚合根建模的一部分，而不是实现阶段的可选优化。聚合根一旦确定，其实体字段中的稳定值域就应同步收敛，避免后续在 `biz / data / service` 中散落裸常量或局部枚举。

具体枚举落位、命名与共享约束，继续参考 `error-enum.md`。

### 实体字段顺序约束

聚合根实体与从属实体定义字段时，默认按以下顺序组织：
1. 普通字段
2. 时间类型字段
3. 关联信息 `Info` 字段
4. 关联信息 `List` 字段

补充约束如下：
- 普通字段优先放稳定主语本体字段，例如 `id`、状态、类型、编号、名称、结果、原因等
- 时间类型字段集中放在普通字段之后，例如 `create_time`、`update_time`、`expire_time`
- 单对象关联信息统一放在 `Info` 区域，避免与本体字段交错
- 列表型关联信息统一放在 `List` 区域，避免与单对象关联字段混排

该顺序用于保持聚合根实体结构稳定、可读，并让“本体字段 / 时间字段 / 关联投影字段”边界清晰一致。

## 组成关系

围绕一个聚合根，通常会同时存在以下几类对象：
- 聚合根本体
  表达稳定业务主语
- 从属实体
  围绕聚合根存在，表达聚合内部从属关系
- 值对象
  表达稳定值语义
- 状态枚举
  表达聚合根或从属实体的稳定状态
- 关系视图
  表达关联对象或关系投影，不伪装成聚合根本体

应用层输入输出对象与 `proto message` 都是聚合根的投影。它们与领域对象是围绕同一聚合根展开的平级产物，不是上下游派生链。

当聚合根或其从属实体需要定义 `json tag` 时，默认使用 `snake_case` 命名，保持领域对象序列化表达稳定一致。

## 组织规则

围绕聚合根组织领域时，应优先遵守以下规则：
- 一个聚合根围绕一个稳定业务对象组织
- 聚合根名称确定后，`repo`、`usecase`、领域文件、应用层对象、`proto` 命名都应围绕它收敛
- 从属实体围绕聚合根语义命名
- 关系视图要明确表达“关系”或“视图”语义，不伪装成聚合根
- 聚合根及其从属实体的 `json tag` 默认使用 `snake_case`
- 对象在跨层传递、上下文传递、事件传递和依赖调用传递时，优先复用语义稳定的现有对象

不要在不同层级为同一业务主语并行发明多套名字。

## 判断提示

判断某个对象是否适合作为聚合根时，可优先观察：
- 它是否表达稳定业务主语，而不是页面或动作
- 它是否能稳定承载状态、关系与领域规则
- 相关 `repo`、`usecase`、领域文件、应用层对象、`proto` 是否都能自然回到它的语义
- 从属实体与关系视图是否都能围绕它组织

如果一个名字只能解释单一页面、按钮动作、临时展示结构或第三方返回体，它通常不适合作为聚合根。

## 示例

### 标准形态

```text
Account
|- 核心字段：id、status、type
|- 从属实体：AccountCollect、AccountFlowPage
|- 关系视图：FirstCheckUser、SecondCheckUser
|- 平级收敛：Account / AccountRepo / AccountUseCase / Account proto
```

### 典型实现

```go
type Account struct {
	ID                  uint32                 `json:"id"`
	Status              openenum.AccountStatus `json:"status"`
	Type                openenum.AccountType   `json:"type"`
	CreateTime          uint32                 `json:"create_time"`
	UpdateTime          uint32                 `json:"update_time"`
	FirstCheckUserInfo  *adminbiz.AdminUser    `json:"first_check_user_info"`
	SecondCheckUserInfo *adminbiz.AdminUser    `json:"second_check_user_info"`
	AccountCollectInfo  *AccountCollect        `json:"account_collect_info"`
	AccountFlowPageList []*AccountFlowPage     `json:"account_flow_page_list"`
}
```
