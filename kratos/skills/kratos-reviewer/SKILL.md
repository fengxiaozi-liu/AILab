# Kratos 代码审查

## 作用

定义“如何审查”，不重复维护架构规则细节。
它负责：

- 如何从代码变化识别审查维度
- 哪些变化模式必须立案调查
- 如何把立案项映射到审查维度
- 如何基于规则类别回查 `kratos-architecture`
- 如何输出结构化 findings
- 尽可能多的一次性审查出来所有不符合知识域事实与规则的问题，做到不遗漏

审查内容本身由以下文档承载：

- `references/checklist.md`：定义审查维度、触发模式、必答问题与风险信号
- `kratos-architecture` 及其 references：提供规则依据与仓库事实来源

## 适用场景

- 新代码生成后检查
- PR / diff 审查
- 架构一致性检查
- 风险识别与问题定位
- 需要结构化 review 结论时

## 不适用场景

- 只要求直接生成代码
- 只做概念解释
- 与 Kratos 无关的普通代码问题

## 事实来源

审查必须基于：

1. 仓库中的实际代码与改动
2. `references/checklist.md` 中定义的审查维度与立案调查项
3. `kratos-architecture` 及其 references 作为规则依据

审查时必须同时遵守 `kratos-architecture` 中对应知识域的事实与规则：

- 变化画像命中的知识域，必须回查对应 `reference` 中的事实说明与 `规则`
- 当前实现只要与对应知识域的规则不符合，就应判定为存在问题
- 不得以“历史代码这样写”“当前能运行”或“局部实现方便”为理由绕过既有规则

## 决策优先级

1. 明确的仓库事实
2. `references/checklist.md`
3. `kratos-architecture` 及其 references
4. 一般 Go / Kratos 工程经验

如果仓库历史实现与架构规则冲突，不得默认历史写法就是正确实现；必须指出它是历史遗留，还是当前仍应遵循的项目约定。

## 工作流

### 第一步：读取变化事实

先看本次审查涉及的代码、diff 或文件，不要一上来逐条套 checklist。
这一阶段只做变化画像，先识别本次改动触达的知识域、结构变化与默认文件级风险，不急于下结论。

优先识别：

- 是否触达 `domain` 主线：业务编排、事务边界、主题 owner、`repo` 抽象、relation、第三方领域依赖、稳定值域进入 `biz`
- 是否触达 `components`：`listener / consumer / crontab / event bus / ent / wire / provider`
- 是否触达 `service`：`proto / service / server / gateway / 协议映射`
- 是否触达共享约束：enum、error、const、跨层共享字面量
- 是否触达 `internal/pkg`：潜在公共能力、跨模块重复 helper、已脱离领域语义的通用函数
- 是否触达同一主题的文件组织方式：聚合根/从属实体、`Repo`、`UseCase` 是否仍收敛在同一业务文件，还是被机械拆成 `xxx.go` / `xxx_usecase.go`

需要关注的变化形式包括：

- 新增、修改、删除、迁移、合并、拆分、重命名
- 调用关系变化
- 事务位置变化
- 注入链路变化
- 状态分支、结果映射、固定值域表达变化
- 第三方 HTTP / SDK 接入或归属变化
- `ent / query / upsert / schema / repo` 组织方式变化

同时对每个涉及文件做必查：

- 是否存在 `nil, nil`、吞错、静默成功或空结果无错误的模糊返回
- 是否存在已脱离领域语义、具备稳定复用价值、应考虑下沉到 `internal/pkg` 的公共函数或公共逻辑
- 构造注入与 `Wire` 已保证的前提，构造注入依赖是否被运行期判空

### 第二步：展开审查维度并回查事实与规则

根据变化画像，对照 `references/checklist.md` 识别本次必须覆盖的审查维度。不要按任务关键词触发审查，也不要把 checklist 当作简单扫表。

对已触发的维度，继续展开对应的必答问题与风险信号，并回到 `kratos-architecture` 中对应知识域的 `reference`，回查事实说明与 `规则`。

当前实现只要与对应知识域的规则不符合，就应判定为存在问题。

若同一改动同时触达多个知识域，应分别回查对应 `reference`，不要只选一份依据。

### 第三步：先形成维度结论，再整理 findings

先按维度给出内部判断，例如：

- 主题与边界：是否偏离
- 分层与职责：是否偏离
- 稳定语义与共享定义：是否偏离
- 注入与装配前提：是否偏离
- data-access 边界：是否偏离

然后只把存在问题的结论展开成 findings。
不要直接把零散观察堆成问题列表；先判断“它违反了哪个审查维度”，再输出问题。

### 第四步：用 checklist 做兜底确认

在输出前，回到 `references/checklist.md`，确认：

- 本次已触发的审查维度都已覆盖
- 没有因为只关注单一问题而遗漏同维度内的其他风险
- findings 都能对应到明确的审查维度与规则依据

checklist 的职责是防漏，不是驱动整个审查流程。

## 高优先级约束

- 不要把审查绑定到单次任务描述、单个关键词或单个文件名
- 不要跳过变化识别，直接套 checklist
- 不要只引用 checklist，不回查架构规则类别
- 不要把代码风格偏好误当成架构问题
- 不要因为仓库里已有历史写法就默认它合理
- findings 必须指出问题类型、位置、证据和原因
- 没有证据时不要武断下结论
- findings 必须区分：架构问题 / 风险问题 / 优化建议
