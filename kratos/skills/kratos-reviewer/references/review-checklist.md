# Review Checklist

## 作用

定义 Kratos 项目代码审查时需要覆盖的检查项、识别提示与典型示例。本文只定义“检查什么”，不定义“如何组织审查流程”。

## 检查项 1：是否按聚合根组织改动

### 检查目标
检查本次改动涉及的文件，是否围绕同一个明确的 aggregate root 组织，目录归属和职责是否一致。

### PASS 标准
- 新增或修改的文件都围绕同一个明确的业务聚合组织
- 目录归属、文件职责、术语命名保持一致
- 没有把专项能力混放到无关聚合根下

### ISSUE 标准
- 同一业务能力被拆散到无关目录
- 某个专项能力被塞进不相关聚合根
- 文件归属与业务语义不一致，导致边界模糊

### 识别提示
- 看本次改动涉及的目录是否仍然围绕同一个业务对象组织
- 看 repo、biz、data、enum、error 是否指向同一个业务语义
- 看是否把专项能力继续堆进通用文件，导致职责漂移

## 检查项 2：命名是否与业务域一致

### 检查目标
检查本次改动中的类型名、函数名、文件名、常量名、错误名，是否以当前聚合根为基础设定，并与该聚合根下的既有术语保持一致。

### PASS 标准
- 命名围绕当前聚合根展开
- 命名与该聚合根下既有术语一致
- 同一概念在当前聚合根内保持统一表达

### ISSUE 标准
- 命名脱离当前聚合根语义
- 混入其他聚合根或其他子域的术语
- 同一概念在当前聚合根内出现多套命名

### 识别提示
- 看命名是否围绕当前聚合根的领域术语展开
- 看同一概念是否在当前聚合根内保持统一命名
- 看名称是否混入其他聚合根或其他子域的语义

## 检查项 3：是否按正确层级落位

### 检查目标
检查本次改动是否按照 Kratos 分层知识落位，各层职责是否清晰，且没有发生职责越层或共享语义错位。

### PASS 标准
- 代码落在职责真正拥有者所在层
- 业务规则、业务语义映射和业务编排没有落到不拥有该职责的层
- 存储实现、下游依赖接入和边界转换没有反向承载业务主流程
- 共享语义没有长期滞留在具体实现层

### ISSUE 标准
- 把业务规则、状态流转或业务语义映射塞进不拥有该职责的层
- 把领域判断写进 repo、handler、listener、server 或其他边界实现中
- 把应进入共享落位的语义长期保留在单个实现文件中

### 识别提示
- 看本次改动里是否新增了业务判断、语义映射、状态流转、规则编排或共享语义定义
- 看这些职责是否落在真正拥有它的层，而不是实现便利的那一层
- 看边界实现、数据实现和业务语义是否发生混写

## 检查项 4：稳定语义是否已收敛

### 检查目标
检查本次改动中的稳定值域、协议字面量、错误语义，是否已经按共享约束完成收敛，并与实现层职责正确分离。

### PASS 标准
- 稳定业务值域已收敛为 enum 或 typed const
- 稳定协议字面量已收敛为具名常量
- 第三方原始值与项目内稳定语义已分层表达
- `not found` 已转为统一错误语义

### ISSUE 标准
- 稳定值域继续以裸 `string` / `int` 长期存在于业务消费链路中
- header、path、query key、content-type、固定动作、固定事件名等稳定协议字面量继续散落在实现文件中
- 第三方原始值未经收敛直接进入业务语义对象或业务消费链路
- `not found` 被处理成 `nil, nil`、空对象成功返回，或模糊错误

### 识别提示
- 看是否出现稳定值域、稳定协议字面量、统一错误语义或第三方边界值
- 看这些值是否跨函数、跨文件、跨层复用
- 看这些值是否进入业务判断、状态流转、结果映射或其他业务消费链路
- 看原始边界值与项目内稳定语义是否已经分层表达

## 检查项 5：是否误放 `internal/pkg`

### 检查目标
检查当前能力是否仍带业务语义，是否被错误地下沉成伪公共能力。

### PASS 标准
- 下沉能力不携带聚合根或业务域语义
- 抽取后的接口和实现可被多个域复用，而不是只为当前业务绕路
- 没有把业务编排、业务错误或领域专有常量塞进 `internal/pkg`

### ISSUE 标准
- 仍带业务语义的能力被包装成公共工具
- 仅为逃避当前目录职责而下沉到 `internal/pkg`
- `internal/pkg` 内出现明显属于单一业务域的术语、错误或协议常量

### 识别提示
- 看该能力脱离当前聚合根后是否仍然成立
- 看它是通用基础设施，还是当前业务的实现细节
- 看下沉后的命名和依赖是否仍然绑定某个业务域

## 检查项 6：Wire / 注入前提是否被运行期兜底

### 检查目标
检查本次改动是否将 `Wire` / provider / constructor 注入的必需依赖，错误地下沉到 `biz / data / service` 方法内做 `nil` 兜底，从而把装配期问题伪装成运行期分支。

### PASS 标准
- 必选注入依赖的校验位于 `NewXxx(...)` 或 provider 阶段，或直接依赖构造前提
- `biz / data / service` 方法内没有针对注入字段的 `nil` 兜底分支
- 没有用普通运行时错误掩盖依赖缺失、配置缺失或装配失败

### ISSUE 标准
- 在 usecase / repo / client / service 方法中出现 `if r.dep == nil { ... }`、`if u.repo == nil { ... }`
- 将“注入依赖缺失”返回为普通 `fmt.Errorf(...)` 或其他运行时错误，而不是在构造期暴露
- 将可选能力与必选注入合并表达，后续再用运行期判空修补

### 识别提示
- 看该依赖是否来自 `NewXxx(...)` 构造函数、`Wire` 或 provider 入口
- 看当前判空分支是在处理外部不可信输入，还是在兜底内部注入前提
- 看错误是否属于“运行时业务失败”，还是实际上是“构造期装配失败”

### PASS 示例
```go
func NewKycRepo(data *kit.Data, tp *conf.ThirdParty) (*kycRepo, error) {
	sumsub := tp.GetSumsub()
	if sumsub == nil {
		return nil, errors.New("...")
	}
	return &kycRepo{data: data, sumsub: sumsub}, nil
}
```

### ISSUE 示例
```go
func (r *kycRepo) sumsubRequest(ctx context.Context) error {
	if r.sumsub == nil {
		return fmt.Errorf("sumsub config is required")
	}
	return nil
}
```

## 检查项 7：是否泄漏 ent / data-access 细节

### 检查目标
检查 `biz` 或更上层是否直接依赖 ent 细节，或把存储实现细节泄漏到业务消费链路。

### PASS 标准
- `biz` 通过仓库抽象消费数据能力
- ent schema、query builder、predicate 等细节停留在 data 层
- 上层只感知业务语义，不感知存储实现细节

### ISSUE 标准
- `biz` 直接依赖 ent 类型、ent 查询细节或存储字段结构
- 将 data-access 细节直接作为上层接口契约暴露
- 业务判断依赖底层存储实现约定而非领域语义

### 识别提示
- 看 `biz` 是否 import 了 ent 相关包或持有 ent 类型
- 看接口返回值是否把底层存储结构直接暴露到上层
- 看业务规则是否依赖 schema 字段细节而不是领域模型

## 检查项 8：runtime 组件是否承载过多业务逻辑

### 检查目标
检查 `consumer / listener / service / crontab` 是否只承担入口、适配和最小转发，而没有错误承载复杂业务逻辑。

### PASS 标准
- runtime 组件只做入口接收、参数转换、调用下层
- 复杂业务判断、状态流转、规则编排留在真正拥有职责的层
- runtime 组件没有演变成隐藏 usecase

### ISSUE 标准
- 把复杂业务流程、状态流转、规则判断直接写进 runtime 组件
- runtime 组件同时承载协议适配和业务主流程
- 为了实现方便把 usecase 逻辑平铺到 service、listener 或 consumer

### 识别提示
- 看 runtime 组件中是否出现大量条件分支、状态判断、规则编排
- 看其职责是“转发入口”还是“实际主流程实现者”
- 看同类逻辑是否本应沉到 biz / usecase
