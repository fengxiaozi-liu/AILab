# Kratos 开发流水线

## 目标

把 Kratos 项目中的结构性改动，按“先识别项目类型、再完成层级定位、再最小实现、最后完成必要验证”的方式闭环完成。

## 适用场景

- 新增功能
- 大型 BUG 修复
- 新增模块
- 涉及多层联动的改动，例如 `proto / biz / data / service / listener / wire / ent`
- 需要判断聚合根、分层归属、协议落位、生成链路的改动

## 不适用场景

- 纯概念性解释
- 纯语法问答
- 明确不涉及结构判断的单点微小修复

## 强制原则

- 先识别当前仓库的项目类型，再决定读取哪些目录和 references
- `kratos-architecture` 是本次结构判断与工程实施的事实源
- `kratos-pipeline` 只负责实施流程，不替代架构裁决，也不覆盖独立的结构确认或审查活动
- 严格按照工作流逐步完成，不允许跳步与缺失步骤

## 工作流

### 1. 识别项目类型

先基于仓库事实判断当前项目属于哪一类，再决定后续读取范围。

至少确认：

- 当前项目属于 `BaseService`、`GatewayService` 还是 `BusinessService`
- 当前项目类型的判断依据是什么
- 本次改动会触达哪些目录和生成链路

输出时必须给出 `项目类型判断`，并说明本次判断依据来自哪些仓库事实。

### 2. 读取仓库事实

- 先读取与当前任务直接相关的目录、现有模块、相似实现、生成链路
- 只做与当前任务相关的扩读，不做无关扩读
- 先确认仓库事实，再做层级定位

### 3. 层级定位

开始实现前，必须先完成本次改动的层级定位。

层级定位应基于仓库事实，并以 `kratos-architecture` 为依据，明确本次改动涉及哪些层以及判断依据。

输出时不要写泛化说明，要直接写清楚“本次改动落在哪几层、依据是什么”。

### 4. 实施最小必要改动

- 实施前以 `kratos-architecture` 为事实源，不得偏移规范
- 只改完成任务所需的最小闭环
- 优先复用现有模式，不并行发明新结构
- 涉及生成链路时，按仓库既有方式生成，不手搓替代物

### 5. 验证闭环

实现后，必须执行与改动范围匹配的验证。

验证要求：

1. 涉及 `proto / wire / ent / provider / listener` 等链路时，必须补齐对应生成或注册验证。
2. 涉及结构性改动时，至少完成构建或定向测试中的一种有效验证；如仓库已有更明确的验证方式，则按仓库方式执行。
3. 若验证失败，不得将当前实现视为完成，必须回到实施步骤修复。
4. 若当前任务除实现验证外还包含独立的结构确认或审查要求，则必须补齐对应确认活动后，才能将当前实现视为完成；`kratos-pipeline` 不代替这类活动本身。

## 最终输出必备章节

最终响应至少应包含以下信息：

- `项目类型判断`
- `层级定位`
- `验证结果`

如果任务包含结构性实现，上述三节缺一不可。

## 停止条件

出现以下情况时，不应继续凭空实现，而应先补齐判断依据：

- 项目类型不明确
- 层级归属不明确
- 仓库现有模式相互冲突
- 生成链路或注入注册链路尚未确认

## 推荐输出骨架

```md
项目类型判断
- 项目类型: ...
- 判断依据: ...

层级定位
- 涉及层级: ...
- 层级依据: ...
- internal/pkg: ...

验证结果
- 已执行: ...
- 结果: ...
- 未执行项: ...
```
