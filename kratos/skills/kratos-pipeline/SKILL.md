# Kratos 开发流水线

## 目标

把 Kratos 项目中的结构性改动，按“先识别项目类型、再完成变化画像与事实回查、再明确实施判断、再做最小实现、最后完成必要验证”的方式闭环完成。

## 适用场景

- 新增功能
- 大型 BUG 修复
- 新增模块
- 涉及多层联动的改动，例如 `proto / biz / data / service / listener / wire / ent`
- 需要判断聚合根、分层归属、协议落位、生成链路的改动

## 不适用场景

- 纯概念性解释
- 纯语法问答
- 明确不涉及结构判断的单点微小修复

## 规则

- 先识别当前仓库的项目类型，再决定读取哪些目录和 `references`
- `kratos-architecture` 是本次结构判断与工程实施的事实源
- 只负责实施流程，不替代架构裁决，也不覆盖独立的结构确认或审查活动
- 严格按照工作流逐步完成，不允许跳步与缺失步骤

## 工作流

### 1. 识别项目类型

先基于仓库事实判断当前项目属于哪一类，再决定后续读取范围。
至少确认：

- 当前项目属于 `BaseService`、`GatewayService` 还是 `BusinessService`
- 当前项目类型的判断依据是什么
- 本次改动会触达哪些目录和生成链路

输出时必须给出 `项目类型判断`，并说明本次判断依据来自哪些仓库事实。

### 2. 读取仓库事实

- 先读取与当前任务直接相关的目录、现有模块、相似实现、生成链路
- 只做与当前任务相关的扩读，不做无关扩读
- 先确认仓库事实，再做实施判断

### 3. 实施前判断

开始实现前，不要直接进入编码，先完成本次改动的变化画像、事实源回查与层级定位。

先识别本次改动触达的 `kratos-architecture` 知识域，以及这些知识域分别回答什么判断问题：

- `aggregate-root`：本次改动围绕哪个稳定业务对象组织，主题是否收敛
- `layer`：代码应落在哪一层、目录职责是什么
- `domain`：业务编排、事务边界、`repo` 抽象、relation、第三方领域依赖如何组织
- `naming`：命名是否围绕稳定主题与聚合根收敛
- `service`：`proto / service / server / gateway` 如何落位，协议与接入层如何组织
- `components`：`listener / consumer / event bus / ent / wire / provider / lifecycle` 如何接入与装配
- `error-enum`：错误语义、稳定值域、协议字面量是否需要收敛
- `pkg`：是否存在公共能力下沉，或不应下沉到 `internal/pkg` 的判断
- `code-style`：实现约束、注入前提、失败路径表达是否符合仓库规范
- `shared-conventions`：日志、注释、共享表达是否需要统一

然后按变化画像回查 `kratos-architecture` 对应事实源，不得跳过事实判断直接凭经验落层。

实施前必须先完成一轮“实施准入检查”，并把结论显式写清楚，不得只做心里判断。至少逐项确认：

- `知识域触达`：本次改动触达了哪些 `kratos-architecture` 知识域；每个知识域分别对应本次要回答什么落层、边界、命名、注入、生成链路或值域问题
- `事实与规则`：对应知识域的仓库事实、现有模式与 `规则` 是否已经明确；若仍说不清“为什么落在这里、为什么不落在别处”，则不得进入编码
- `模糊返回排查`：准备写入的文件中，不存在 `nil, nil`、吞错、静默成功、空结果无错误、只记日志不返回错误等模糊失败表达；若需要忽略错误，必须能给出明确的事务或组件语义依据
- `internal/pkg 下沉判断`：写入代码中凡是不承载业务语义、且具备稳定复用价值的部分，必须先判断是否应下沉到 `internal/pkg`；不得把“目前只用一次”当作拒绝下沉的理由；反过来，带业务语义的逻辑也不得伪装成公共能力下沉
- `注入前提确认`：凡属构造注入、`Wire`、provider 已保证成立的依赖，禁止在运行期追加判空式防御代码；只有当仓库事实表明该依赖允许缺省、延迟装配或可选注入时，才允许显式判空
- `实施闭环确认`：当前准备是否已经能够回答代码落层、职责边界、依赖方向、生成链路、注册链路与最小闭环范围；如果这些问题中任一项仍不明确，应先回到事实源补足判断，再继续实施

可以把这轮检查压缩为一个简短小节输出，但不能省略判断本身。若上述任一项无法回答清楚，不要直接进入编码。

### 4. 实施最小必要改动

- 实施前已确认的层级定位、边界与生成链路，不得在编码过程中随意漂移
- 以 `kratos-architecture` 为事实源，不得偏移规范
- 只改完成任务所需的最小闭环
- 优先复用现有模式，不并行发明新结构
- 涉及生成链路时，按仓库既有方式生成，不手搓替代物
- 如果实现过程中发现原判断不足，应先回到事实源修正层级定位，再继续编码
- 不得把“顺手可做”的结构扩张混入本次最小闭环

### 5. 验证闭环

实现后，必须执行与改动范围匹配的验证。

- 涉及 `proto / wire / ent / provider / listener` 等链路时，必须补齐对应生成、注册或构建验证
- 涉及结构性改动时，至少完成一种有效验证，如定向测试、构建或仓库既有验证方式
- 若验证失败，不得将当前实现视为完成，必须回到实施步骤修复
- 负责实现闭环；若任务还要求独立审查或结构确认，应另行完成

## 最终输出必备章节

最终响应至少应包含以下信息：

- `项目类型判断`
- `实施前判断`
- `验证结果`

如果任务包含结构性实现，上述三节缺一不可。

## 停止条件

出现以下情况时，不应继续凭空实现，而应先补齐判断依据：

- 项目类型不明确
- 层级归属不明确
- 仓库现有模式相互冲突
- 生成链路或注入注册链路尚未确认

## 推荐输出骨架

```md
项目类型判断
- 项目类型: ...
- 判断依据: ...

实施前判断
- 涉及知识域: ...
- 实施准入结论: ...
- 最小闭环: ...

验证结果
- 已执行: ...
- 结果: ...
```
