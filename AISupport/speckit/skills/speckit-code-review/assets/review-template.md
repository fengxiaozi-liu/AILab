# 审查报告模板

在 `speckit-code-review` 输出 `review.md` 时，优先按以下结构组织内容。

```md
# Review

## 全局问题列表

### 未解决问题

- [ ] REVIEW-001 [P1 | 完成度 | 位置: 文件/模块] 问题摘要

### 已解决问题

- [✅️] REVIEW-002 [P2 | 架构与规范 | 位置: 文件/模块] 问题摘要

## 审查模式与上下文

- 审查模式: [Feature review | Project-aware review | General review]
- 审查范围: [本次审查覆盖的 feature、模块、提交范围或文件]
- 已加载的项目知识/项目 skill:
  - [名称]
- 审查输入:
  - [spec.md / plan.md / tasks.md / 代码 / 测试 / 配置]

## 项目知识与规范依据
- [本次判断所依据的项目知识、规范、既有约束、仓库模式]
- [若无明确项目规范，说明退化为通用审查基线]

## 完成度
- 结论: [通过 / 存在问题]
- 覆盖说明: [需求、任务、代码、测试、文档之间的闭环情况]
- 待修复问题:
  - 优先级: [P1 | P2 | P3]
  - 位置: [文件/模块]
  - 问题: [缺失内容或未闭环项]
  - 依据: [spec / plan / tasks / 项目知识 / 仓库模式]
  - 建议: [最小修复方向]

## 正确性
- 结论: [通过 / 存在问题]
- 覆盖说明: [主流程、失败路径、边界情况、状态流转]
- 待修复问题:
  - 优先级: [P1 | P2 | P3]
  - 位置: [文件/模块]
  - 问题: [错误行为或高风险缺陷]
  - 依据: [代码行为 / 测试 / spec / 项目知识]
  - 建议: [修复方向]

## 架构与规范
- 结论: [通过 / 存在问题]
- 覆盖说明: [分层、依赖、抽象、命名、项目规范符合度]
- 待修复问题:
  - 优先级: [P1 | P2 | P3]
  - 位置: [文件/模块]
  - 问题: [架构偏差、规范违背、模式不一致]
  - 依据: [项目知识 / 项目规范 / 仓库既有模式]
  - 建议: [调整方向]

## 性能

- 结论: [通过 / 存在问题]
- 覆盖说明: [高开销操作、阻塞、读写模式、并发模型]
- 待修复问题:
  - 优先级: [P1 | P2 | P3]
  - 位置: [文件/模块]
  - 问题: [性能风险]
  - 依据: [代码路径 / 数据访问模式 / 工作负载假设]
  - 建议: [优化方向]

## 安全

- 结论: [通过 / 存在问题]
- 覆盖说明: [输入校验、鉴权、敏感数据、注入面]
- 待修复问题:
  - 优先级: [P1 | P2 | P3]
  - 位置: [文件/模块]
  - 问题: [安全风险]
  - 依据: [输入面 / 权限模型 / 日志 / 配置]
  - 建议: [修复方向]

## 假设与未知项

- [当前审查成立所依赖的假设]
- [尚未验证，但可能影响结论的未知项]

## 结论与下一步
- 审查结论: [通过 / 需修复后复审]
- 下一步建议:
  - [若有高优先级问题，先修复后重新/code-review]
  - [若通过，进入/summary]
```

要求：
- 模板开头必须包含 `全局问题列表`，并按 `未解决问题` 与 `已解决问题` 分组
- 全局问题列表中的问题编号统一使用 `REVIEW-xxx`
- 未解决问题统一使用 `- [ ]`
- 已解决问题统一使用 `- [✅️]`
- 主体目录必须按 `完成度 -> 正确性 -> 架构与规范 -> 性能 -> 安全` 展开
- 每个目录都必须显式给出 `结论` 与 `覆盖说明`
- 若某个目录无问题，`待修复问题` 写 `- 无`
- 每条待修复问题都必须显式标注 `优先级`
- 优先级统一使用 `P1 / P2 / P3`
- 每条问题都应尽量写明判断依据，优先引用 `spec`、`plan`、`tasks`、项目知识、仓库既有模式
- 若本次属于 `Project-aware review`，必须显式写出所采用的项目知识或项目规范
- 若本次没有可靠项目知识，必须在 `项目知识与规范依据` 中明确说明退化为通用审查维度
