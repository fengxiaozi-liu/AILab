# Spec Kit Specify Skill

## 何时使用

- 用户给出自然语言需求，希望生成首版 `spec.md`
- 已有零散想法或旧草稿，需要整理成结构化 spec
- **不用于**回答已有 CQ；已有 CQ 时应优先交给 `/clarify`

## 职责边界

- `speckit-specify` 只负责把输入收敛为**首版规格文档**
- 可以生成初始 CQ，但**不负责**和用户来回完成澄清闭环
- 不产出持久化 checklist 文件；需求质量审计交给 `/checklist`

## 输入

- 必需：用户需求描述
- 可选：已有 `specs/<feature>/spec.md`（用于增量整理）
- 模板：本技能目录下 `./templates/spec-template.md`

## 工作流

### 1. 上下文获取

1. 按需读取与当前需求直接相关的仓库上下文（如 `README.md`、已有 `specs/`、必要的架构文档, 仓库代码实现信息）。
2. 如果发现已有同名或高度重叠的 feature，优先复用或增量更新，而不是重复新建。

### 2. 短名生成

从用户描述生成 2-3 词 feature 短名：

- 从描述提炼关键动作和对象
- 优先使用“动词-名词”风格
- 保留必要领域术语或缩写
- 全小写，单词间用 `-` 连接

### 3. 首版需求提炼

1. 读取本技能目录下 `./templates/spec-template.md`。
2. 在开始生成前，明确说明本次输出将使用本技能目录下 `./templates/spec-template.md` 作为结构骨架。
3. 严格按模板的主章节组织内容，再填入本次需求的真实信息。
4. 从用户输入中提炼并写入：
   - 背景与动机
   - 目标用户
   - 核心需求（RQ）
   - 关键实体
   - 约束与边界
5. 约束：
   - 只写“要什么”和“为什么”，不写实现方案
   - 每条 RQ 必须可测试、可追踪、尽量无歧义
   - 无法确认的信息不要擅自拍板

### 4. 初始 CQ 生成

当输入中仍存在会影响后续设计或任务拆解的缺口时，生成初始 CQ：

- 仅生成真正影响架构、数据建模、边界定义、验收口径的问题
- 编号使用 `CQ-001`, `CQ-002` ...
- 每个 CQ 必须包含：
  - 类别
  - Q (提问)
  - 可选项（优先提供 2-4 个互斥候选，使用 `A.`, `B.`, `C.`, `D.` 编号；若无法给出可靠候选，则不新增选项）
  - A (澄清结论) = `[待澄清]`
- 能枚举常见决策分支时，必须补出可选项，避免只给开放式问题
- 可选项要覆盖主要方案与边界取舍，措辞简短、可直接回复，不要写成实现细节
- 若存在任何未解 CQ，则将文档状态设为 `Clarifying`
- 若无未解 CQ，则将文档状态设为 `Ready`

### 5. 轻量自检

生成前做一次就地质量检查：

- 需求描述不含代码、框架、数据库或 API 设计细节
- 每条 RQ 都能推导出验收条件
- 关键实体和边界条件没有明显遗漏
- CQ 只覆盖真正未决的问题，不堆砌形式化追问

## 输出

- `specs/<feature>/spec.md`
- 若存在未解决 CQ，明确提示用户调用 `/clarify` 完成澄清闭环
- 若文档已是 `Ready`，提示用户先调用 `/checklist` 做需求质量门检查，再进入 `/plan`
