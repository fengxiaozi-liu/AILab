# Spec Kit Tasks

## 何时使用

- 需要把计划拆成可执行任务时使用
- `plan.md` 存在时优先基于 plan 生成
- 若缺少 plan，则基于用户输入和可选 spec 生成最小任务清单

## 输入

- 必须：用户输入（范围、约束、交付物）
- 可选：`specs/<feature>/plan.md`
- 可选：`specs/<feature>/spec.md`
- 模板：本技能目录下 `./templates/tasks-template.md`

## 职责边界与依赖

- `speckit-tasks` 负责输出具备清晰组织结构与技术标识的 `tasks.md`
- 该产物直接服务于后续 `speckit-implement`
- 任务编排应兼顾实施顺序、依赖隔离、认知负荷和可验证性
- 若已识别到项目类型或项目约束，应以对应项目规则为准

## 工作流

### 1. 前置检查

1. 读取本技能目录下 `./templates/tasks-template.md`，将其作为输出结构基准。
2. 明确本次 `tasks.md` 基于模板生成。
3. 获取当前需求的基础上下文。
4. 若存在 `spec.md`，提取 RQ 编号列表。
5. 若存在 `plan.md`，提取阶段划分、架构设计要点与潜在风险。
6. 识别当前项目类型；若有特定项目约束，将其纳为任务排布基准。
7. 若缺少足够文档依据，则基于最原始输入推导最小任务清单，并标注风险。
8. 加载工程项目需要的知识域或 SKILL，作为本次任务创建的依据。
9. 对本次任务所属的软件开发领域和技术进行判定（例如测试-黑盒测试+白盒测试），从本地加载相关SKILL，并整理为可写入 `tasks.md` 头部的“技术链依赖”说明。

### 2. 上下文与依赖提取

从已有材料中提取：

- 核心数据流与调用链路
- 物理阻塞节点，例如表结构、模型、核心服务、接口、前端接入
- 技术切面边界，例如 DB、逻辑层、接口层、视图层、配置层
- 各 Phase 对应的技术链或 SKILL 依赖关系，例如某一阶段依赖的系列 skill、知识域或执行能力

### 3. Phase 划分原则

Phase 的存在不仅是为了串行防冲突，也是为了给后续 `speckit-implement` 提供更低认知负荷、更高内聚的实施上下文。

划分时优先遵守以下原则：

1. 优先隔离强依赖。存在明确时序依赖时，优先拆分为前后 Phase。
2. 优先保持高内聚。即使可以并行，也要避免把上下文撕裂严重的任务混进同一实施批次。
3. 控制单个 Phase 容量。一个 Phase 内任务过多时，应按业务模块或技术切面继续细分。

同时必须遵守以下执行编排规则：

- 跨 Phase 强制隔离：有强依赖关联的任务，不得放入同一 Phase。
- 同级任务极致压平：同属于一个前置依赖层级的多项无依赖任务，必须全部收敛到同一个 Phase 内，不得人为打散到多个后续 Phase。
- 默认按 Phase 顺序逐阶段串行阻塞。
- 执行器在进入某个 Phase 时，应全权处理该组内的所有任务包，可自行决定并发实施或依次实施。
- 若已识别到项目约束，应以对应项目规则为准。

### 4. 任务生成粒度与排布

1. 粒度控制：每个任务应做到可执行、可验证、可闭环，不要过大，也不要细碎到行级修改。
2. 扁平排布：在单个 Phase 内，直接以列表列出该阶段任务。
3. 精确路径与意图标注：每个任务都应显式给出技术域标签与精确路径或产物。

### 5. 任务记录格式

- 所有任务行的格式以本技能目录下 `./templates/tasks-template.md` 中的示例为唯一基准。
- 模板负责最终产物结构，不在模板正文中承载生成规则说明。
- 在 `Created`、`项目识别结果` 之后，允许增加“技术链依赖”区块，用于说明本次任务所依赖的技术链、系列 skill 或知识域及其与各 Phase 的对应关系。
- 若已识别出明确的技术链或 skill 链路，则必须写入该区块；若未识别，则使用 `未识别`，不要省略该字段。

### 6. 写入规则

- 写入目标文件：`specs/<feature>/tasks.md`
- 以本技能目录下 `./templates/tasks-template.md` 作为结构骨架，将占位符替换为真实内容
- 结构采用自顶向下的 Phase 铺排，各阶段天然自带前置顺序依赖
- 若存在技术链或 SKILL 依赖，则将其写入 `项目识别结果` 后的 `技术链依赖` 字段；推荐写法为按 Phase 说明依赖关系，例如：`Phase 1 使用 xxx，Phase 2 使用 xxx，Phase 3 使用 xxx`

### 7. 质量校验维度

| # | 校验项 | 标准 |
|---|--------|------|
| 1 | 需求覆盖无死角 | 每条需求点至少映射到一个实体 TaskID |
| 2 | 技术域解耦 | 跨度极大的不同技术栈任务是否被有意识分配到合适的 Phase |
| 3 | 防爆栈控制 | 单一 Phase 内的任务数量是否过多，是否做了适时拆分 |
| 4 | 上下文自洽 | 任务间先后顺位是否合理 |
| 5 | 定位精确 | 是否提供了充分的相对路径指引或组件目标标识 |

## 输出

- `specs/<feature>/tasks.md`
- 完成后提示用户复查 `tasks.md`，确认无误后衔接执行 `/implement`
