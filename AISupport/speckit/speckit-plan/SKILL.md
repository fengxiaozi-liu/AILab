# Spec Kit Plan Skill

## 何时使用

- 需要生成结构化的技术实施计划 `plan.md`
- 已有 `spec.md`，需要把业务需求转成技术落地方案
- 没有完整 spec，但亟需一份技术预判与架构层级梳理草案

## 输入

- 必需：用户输入，包括业务目标、范围、核心约束、预期交付物
- 可选：`specs/<feature>/spec.md`
- 模板：本技能目录下 `./templates/plan-template.md`（工作流开始前优先读取，输出尽量沿用其章节结构与表格格式）

## 职责边界

- `speckit-plan` 负责在正式拆分任务条目前，做好宏观的领域调研、技术选型、模块化设计与上下游风险识别。
- 应尽量在输出中为下游任务拆解指明技术层界限和主要依赖链。

## 工作流

### 1. 前置检查与推演

1. 读取本技能目录下 `./templates/plan-template.md`。
2. 在输出前明确说明本次 `plan.md` 将基于本技能目录下 `./templates/plan-template.md` 生成。
3. 将该模板作为本次输出的结构骨架，再填入推导结果。
4. 如果存在 `specs/<feature>/spec.md`，提取需求、关键实体边界。
5. 如果 spec 不存在或信息不足，继续生成草案，但显式列出待澄清项。
6. 识别当前项目类型和目录结构，以便加载与项目匹配的规则。

### 2. Phase 0 项目识别与领域调研

在深入细节设计前，建议先明确环境栈以及业务的核心交互边界：

1. **识别项目类型与关联 Skill**：优先基于代码库事实判断当前工程架构（如前端 Vue/React、后端 Node/Go 等）。在梳理计划时，预判本次需求是否会明显依赖其他框架级或语言专属 Skill，并在后续设计与任务中体现这些约束。
2. **分析关键实体及边界拓扑**：
   - **全栈/后端视角**：涉及哪些核心数据模型（Model）、物理表结构及外键关系的变动？
   - **前端/视图视角**：涉及哪些核心数据容器（Store/Context）、复合组件树、路由规划或高阶页面聚合？
3. **分析状态流转与异常路径**：业务主链路或用户核心交互流是什么？哪里可能面临网络断档、数据竞态或 UI 的异常熔断保底？
4. **识别外部依赖与防阻塞边界**：梳理有哪些模块开发是强互斥或强绑定的，明确上下游的**强依赖阻塞链**（例如接口契约未定可能阻塞 UI 联调），为下游 tasks 的隔离提供依据。
5. 对存在明显取舍的调研项，优先输出：
   - 决策（Decision）
   - 理由（Rationale）
   - 被排除的备选方案（Rejected Alternatives）

### 3. Phase 1 技术设计

在此环节将前期的领域调研落地为具体技术指引。根据识别到的当前项目类型输出设计要点：

通用设计视角示例：

- **服务端项目**：核心实体、接口契约边界、数据访问中间件、测试与验证
- **库 / SDK 项目**：公共接口签名、兼容性、依赖边界、示例与测试
- **CLI 工具**：命令结构、参数输入、输出格式、错误处理、执行流程
- **前端应用**：页面树拓扑、状态管理、组件拆分树（父子组件依赖）、接口联调点、测试策略
- **网关 / 接入层项目**：路由、代理、协议适配、依赖边界
- **数据处理项目**：输入输出格式、批处理流程、容错策略、监控与验证

**为下游 Tasks 铺垫：**
在技术设计中，尽量显式梳理本次需求涉及的技术面（例如：DB 层、核心逻辑层、前端视图层），帮助下游 `speckit-tasks` 做更清晰的拆分。

### 4. 写入规则

- 输出路径：`specs/<feature>/plan.md`
- **优先沿用模板格式**：尽量保留本技能目录下 `./templates/plan-template.md` 的标题层级、表格结构和区块；如有少量不适用内容，可在不破坏骨架的前提下调整。
- Phase 0 / Phase 1 的各设计点必须可追溯到 `spec`（如有）的需求点。
- 如果 spec 缺失导致无法敲定数据结构，请在 plan 中显式预留“【待定 - 需用户澄清】”的高亮占位符。

### 5. 交接质量校验

| # | 校验项 | 标准 |
|---|--------|------|
| 1 | 实体与流转 | 关键数据结构和流转状态是否已进行决策并论述理由？ |
| 2 | 技术切面 | 是否明确划分了前端、后端、基座等不同技术视角下的组件设计？ |
| 3 | 依赖明晰 | 调研中是否明确论述了核心模块间的阻塞依赖关系？ |
| 4 | 架构覆盖 | Phase 1 的所有技术架构设计是否都覆盖了核心需求？ |
| 5 | 无断头路 | 是否留有明显的逻辑黑洞未填补？ |

## 输出动作

以本技能目录下 `./templates/plan-template.md` 作为输出的**结构骨架**：保留其章节层级与表格结构，将所有占位符替换为本次推导出的真实内容。如有需要可在既有章节**内部**追加内容，但不应删减模板已定义的任何章节或字段。

- 生成落盘 `specs/<feature>/plan.md`
- 提示用户 review 架构与依赖关系。确认无明显硬伤后，提示其可调起 `/tasks` 进行清单级拆算。
